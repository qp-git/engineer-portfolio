# OdaiBox 設計判断メモ

OdaiBox は、Discordで通話しながらゲームをするコミュニティ向けに、お題をランダムに出すBotです。

このページでは、単にBotを動かすための技術説明ではなく、なぜその構成にしたのかを、利用体験・運用面・管理面の3つに分けて整理します。

## 1. 利用体験：コミュニティに合わせて遊び方を変えられること

OdaiBoxの主目的は、Discordで通話しながらゲームをするときに、遊び方のマンネリ化を減らすことです。

通常の遊び方が続くと、どうしても同じ流れになりやすくなります。そこで、毎回人が考えるのではなく、Botがランダムにお題を出すことで、遊び方に変化をつけられるようにしました。

また、利用者が最初からオリジナルのお題を考えて遊ぶのはハードルが高いと考え、デフォルトのお題を用意しました。さらに、たまにしか出てこないレアなお題も設計し、導入直後からコンテンツとして楽しみやすいようにしました。

一方で、開発者が用意したお題だけでは、使い続けるうちに飽きが来やすくなります。また、利用者の要望を受けてお題を追加・変更するたびに、開発者がコードを修正してデプロイする運用も現実的ではありません。

そこで、利用者側でDiscordサーバー（コミュニティ）の雰囲気に合わせて、お題を追加・編集・ON/OFFできるWeb管理画面を用意しました。

この判断により、Botを作った本人だけが管理するのではなく、実際に遊ぶ人たちが自分たちのノリに合わせてBotを育てられる構成にしました。
## 2. 運用面：要件に合わせて運用費を抑えること

OdaiBoxは、常時メッセージを監視し続けるBotではなく、Discordのスラッシュコマンドが実行されたときだけ処理できればよいサービスです。

そのため、常時起動のEC2ではなく、API Gateway + Lambda を中心にした構成が合っていると判断しました。

実際に `/odai` や `/admin-login` が実行されたときにAPI Gateway経由でLambdaを呼び出し、必要な処理だけを実行しています。

管理画面も、S3 + CloudFrontで静的ファイルとして配信し、お題の保存・編集処理だけをAPI Gateway + Lambdaに送り、DynamoDBに保存する構成にしました。

DynamoDBは、Discordサーバー（コミュニティ）ごとにお題データや一時ログイン情報を保存するために利用しました。サーバーIDをキーにしてデータを取得する用途が中心だったため、シンプルなKey-Value型の構成と相性がよいと判断しました。

このように、利用頻度や処理内容に合わせて構成を選ぶことで、常時稼働するサーバーを持たずに運用できる形にしました。

結果として、友達に自由に使ってもらっても許容範囲に収まる運用費で、Discordコミュニティ向けのBotを運用できる構成にできました。

## 3. 管理面：操作権限と保存情報を明確にすること

Web管理画面を用意するとお題の編集が可能になる一方で、誰が編集できるのかを考える必要があります。

管理画面のURLを知っているだけで誰でもお題を編集できる状態にはしたくなかったため、URLとパスワードの扱いが課題になりました。

そこで、Discordの `/admin-login` コマンドを入口にして、権限のあるユーザーに管理画面URLと一時パスワードを発行する方式にしました。

固定の共通パスワードを配布する方式では、パスワードの共有・流出・変更が課題になります。

一時パスワード方式にすることで、長期間使い回される固定パスワードを避けられます。また、Discord側から発行することで、DiscordサーバーIDと対象の管理画面を結びつけられる構成にしました。

この方式は便利で運用しやすい一方で、OdaiBoxはDiscordサーバーIDやユーザーIDを扱います。

これらはパスワードのような秘密情報ではありませんが、利用者やコミュニティを識別できる情報です。

そのため、運営する側として、保存する情報と保存しない情報を整理し、プライバシーポリシーとして明文化しました。

特に、音声通話の内容、チャット本文、DM、Discordのパスワードは保存しない方針にしています。

## このプロジェクトで学べたこと

このプロジェクトを通して、Bot本体を作るだけでなく、利用者が操作するWeb管理画面を用意すると、設計で考える範囲が広がると学びました。

特に大きかった学びは以下です。

- 利用者が自分たちのコミュニティに合わせてBotをカスタムできる設計にすること
- お題をコード内に固定せず、DynamoDBと管理画面で変更できるようにすること
- Discordサーバー（コミュニティ）ごとにデータを分けて扱うこと
- 管理画面を用意する場合、操作権限をどう確認するか考えること
- Discord IDなどの識別情報を扱う場合、保存する情報と保存しない情報を説明できるようにすること
- 便利な機能を追加すると、認証・認可・プライバシー説明も設計対象になること

## 今後の改善余地

今後改善するなら、以下を検討します。

- 権限ロールのより細かい制御
- エラー時の管理者通知
- 一時ログイン情報やお題履歴の保存期間の整理
- DynamoDBのキー設計やTTL設定の見直し
- 管理画面APIのアクセス制御強化
- CORSやレート制限など、管理画面APIまわりの保護
