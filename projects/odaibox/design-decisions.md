# OdaiBox 設計判断メモ

OdaiBox は、Discord のボイスチャット中に使う「お題」をランダムに出す Bot です。

単に Bot を動かすだけでなく、非エンジニアでもお題を管理できること、低コストで運用できること、不要な情報を保存しないことを重視して設計しました。

## Lambda中心のサーバーレス構成にした理由

OdaiBox は、常時メッセージを監視し続ける Bot ではなく、Discord のスラッシュコマンドが実行されたときだけ処理できればよいサービスです。

そのため、常時起動の EC2 ではなく、API Gateway + Lambda を中心にした構成にしました。

この構成にした理由は以下です。

- 小規模利用ではコストを抑えやすい
- サーバーの常時運用やOS管理を減らせる
- コマンド実行時だけ処理すればよい用途に合っている
- 学習目的として、イベント駆動構成を体験できる

一方で、Lambda にはコールドスタートや実行時間制限があります。

OdaiBox では、処理内容が軽く、長時間の常時接続も不要なため、この制約は大きな問題にならないと判断しました。

## DynamoDBを使った理由

お題データは、Discord サーバーごとに分けて保存する必要があります。

また、RDB のような複雑な JOIN やトランザクションよりも、サーバーIDをキーにしてお題一覧を取得するようなアクセスが中心です。

そのため、DynamoDB を利用しました。

DynamoDB を選んだ理由は以下です。

- Lambda と相性がよい
- サーバー管理が不要
- サーバーIDをキーにしたデータ取得に向いている
- 小規模利用では低コストで運用しやすい
- AWSのマネージドサービス構成を学習できる

RDB と比べると、柔軟な検索や複雑な集計には向いていません。

ただし、OdaiBox では「サーバーごとのお題を取得する」「お題を追加・編集する」「一時ログイン情報を確認する」といった単純なアクセスが中心のため、DynamoDB の方が構成に合っていると判断しました。

## Web管理画面を用意した理由

当初、Bot のお題はコードやJSONで管理することも考えられます。

しかし、それだと非エンジニアの利用者が自分でお題を追加・編集できません。

OdaiBox では、ゲーム仲間が自分たちでお題を調整できることを重視し、Web管理画面を用意しました。

この判断により、以下を実現できます。

- コードを触らずにお題を追加できる
- お題の有効 / 無効を切り替えられる
- サーバーごとにお題を管理できる
- Botを使う人自身が運用に参加できる

管理画面は S3 + CloudFront で静的配信し、保存・編集処理だけ API Gateway + Lambda に送る構成にしました。

表示部分と処理部分を分けることで、常時起動のWebサーバーを持たずに管理画面を提供できます。

## `/admin-login` による一時パスワード方式にした理由

管理画面をURLだけで編集できる状態にすると、URLを知っている人なら誰でもお題を変更できてしまいます。

一方で、固定の管理者パスワードを配る方式にすると、パスワードの共有・流出・変更が課題になります。

そこで、Discord の `/admin-login` コマンドを使い、権限のあるユーザーだけに一時パスワードを発行する方式にしました。

この方式にした理由は以下です。

- Discord側の権限確認を利用できる
- 固定パスワードを配布しなくてよい
- 一時パスワードに有効期限を設定できる
- サーバーごとに管理対象を分けやすい
- 管理画面の編集権限をDiscord運用と紐づけられる

この設計により、管理画面を便利にしつつ、最低限のアクセス制御を行えるようにしました。

## 保存する情報を絞った理由

OdaiBox は、音声通話中に使う Bot ですが、音声通話の内容を保存する必要はありません。

また、通常のチャット本文やDMを保存する必要もありません。

そのため、保存する情報は機能提供に必要な範囲に限定しました。

保存する主な情報は以下です。

- Discord サーバーID
- Discord ユーザーID
- お題データ
- お題の有効 / 無効状態
- 一時ログイン情報
- お題履歴

一方で、以下は保存しない方針にしました。

- 音声通話の内容
- チャット本文
- DM
- Discord のパスワード
- メールアドレス
- 支払い情報

Discord ID は秘密情報ではありませんが、利用者やサーバーを識別できる情報です。

そのため、必要な範囲に絞って保存し、プライバシーポリシーとして明文化しました。

## プライバシーポリシーを用意した理由

身内向けの小さな Bot であっても、利用者のIDやサーバーIDを扱う以上、何を保存して何を保存しないのかを説明できる状態にしておくべきだと考えました。

特に、Discord Bot は「チャットを読んでいるのではないか」「音声を保存しているのではないか」と誤解される可能性があります。

そのため、以下を明文化しました。

- 保存する情報
- 保存しない情報
- 情報の利用目的
- 管理画面ログインの仕組み
- 一時パスワードの扱い
- 第三者提供しないこと

この整理により、利用者への説明だけでなく、設計上の境界も明確になります。

## GitHub Actionsでデプロイを自動化した理由

Lambda や静的ファイルを手作業で更新すると、更新漏れや手順ミスが起きやすくなります。

そのため、GitHub Actions を使って、main ブランチへの反映をきっかけにデプロイできる構成にしました。

自動化した理由は以下です。

- 手作業による更新漏れを減らす
- Lambda と管理画面の更新手順を統一する
- 変更履歴をGitHubに残せる
- 小さな修正を反映しやすくする
- 実務に近いCI/CDの流れを学習する

個人開発でも、変更から反映までの流れを整えることで、運用しやすさが上がると学びました。

## 今後の改善余地

現時点の構成は、小規模なDiscordサーバーで使うことを前提にしています。

今後改善するなら、以下を検討します。

- 管理操作ログの見やすい表示
- お題データのエクスポート / インポート
- サーバーごとの設定画面の整理
- 権限ロールのより細かい制御
- エラー時の管理者通知
- DynamoDBのデータ設計の見直し
- CloudFrontやAPI Gatewayのアクセス制御強化

学習プロジェクトとしては、機能を増やすことよりも、利用者が安心して使える設計と、説明できる構成にすることを重視しました。
