# Interview Knowledge Bridge 設計判断メモ

## このドキュメントの目的

このドキュメントでは、Interview Knowledge Bridge で行った設計判断を整理します。

README ではプロジェクト概要を、[architecture.md](./architecture.md) では Custom GPT、Bridge API、ホワイトリストの関係を説明しています。ここでは、なぜ中継 API にしたか、なぜホワイトリスト制御にしたか、AI に渡す情報の範囲をどのように考えたかを中心にまとめます。

## 1. PublicとPrivateで情報の役割を分けた理由

このプロジェクトでは、人間向けに公開する情報と、AI に補助情報として渡す情報を分けて考えました。

公開ポートフォリオには、プロジェクト概要、構成、設計判断、学びなど、初見の人が読みやすいように整理した情報を置きます。この公開ポートフォリオリポジトリが、その役割にあたります。

一方で、より細かいコード解説、制作背景、判断理由、作業中の補足メモなどは、公開ページには少し細かすぎる場合があります。しかし、AI に技術整理を補助させる場合には、こうした詳細情報が回答材料として役立つことがあります。

そのため、詳細な補足情報は Private リポジトリ側で管理し、必要なものだけを AI に参照させる方針にしました。

## 2. Privateリポジトリを直接読ませず、中継APIにした理由

Custom GPT から Private リポジトリ上の Markdown を参照する場合、直接ファイルを読ませる方法も考えられます。

しかし、直接参照にすると、どの文書を読めるのか、どの形式で返すのか、将来どのように対象文書を増やすのかを管理しづらくなります。

そこで、Custom GPT と Private GitHub リポジトリの間に Bridge API を置く構成にしました。

中継 API にすることで、以下を整理できます。

- Custom GPT が呼び出す入口を固定できる
- 文書 ID と GitHub 上のパスの対応を API 側で管理できる
- ホワイトリストによって参照範囲を制御できる
- 不要なファイルや古いメモを返さないようにできる
- 将来的に保存先や文書構成が変わっても、GPT 側の呼び出し方を大きく変えずに済む

このように、中継 API は単なるファイル取得処理ではなく、AI に渡す情報の入口を管理する役割を持っています。

## 3. ホワイトリストで参照対象を制御した理由

Private リポジトリには、AI に渡したい情報だけでなく、作業途中のメモや古い試行錯誤も残る可能性があります。

そうした情報まで AI が参照すると、回答に不要な情報が混ざったり、現在の方針と違う内容を拾ったりする可能性があります。

そこで、Custom GPT が取得できる Markdown をホワイトリストで制御する構成にしました。

ホワイトリストを使うことで、以下を明確にできます。

- AI に参照させてよい文書
- 回答生成に使ってよい補足情報
- 現時点で有効なメモ
- プロジェクトごとの参照対象
- 参照対象から外すべき古いメモやノイズ

この設計により、AI に渡す情報の範囲を明示的に管理できるようにしました。

## 4. 中継APIの設計がホワイトリスト制御に影響すると考えた理由

このプロジェクトでは、ホワイトリストを用意するだけでは不十分だと分かりました。

たとえば、Bridge API が任意の GitHub URL やファイルパスをそのまま受け取る設計だと、ホワイトリストに登録していないファイルを指定できる余地が残ります。その場合、ホワイトリストは一覧として存在していても、実際の参照制御としては機能しません。

そのため、Custom GPT からは任意のパスではなく document_id を受け取り、API 側で document_id と GitHub パスの対応を管理する設計にしました。

この設計により、以下を実現しやすくなります。

- Custom GPT に任意の GitHub パスを指定させない
- ホワイトリストにある document_id だけを受け付ける
- GitHub 上の実際のパスは API 側で管理する
- Markdown 以外のファイルを参照対象にしない
- 許可していない文書は API から返さない

この経験から、セキュリティや参照制御は、単に許可リストを作るだけでなく、API の入力設計やデータ取得経路まで含めて考える必要があると学びました。

## 5. Markdownを参照対象にした理由

参照対象は Markdown としました。

Markdown は、人間が直接読んでも分かりやすく、GitHub 上でも管理しやすい形式です。また、見出しや箇条書きで情報を整理しやすいため、AI に渡す補助コンテキストとしても扱いやすい形式です。

ポートフォリオ整理では、プロジェクトごとに以下のような情報を残しやすくなります。

- 概要
- 構成
- 設計判断
- 詰まった点
- 改善案
- 補足説明
- 回答生成の材料

Markdown にすることで、人間が編集しやすく、AI も参照しやすい情報管理ができます。

## 6. AIに渡す情報の範囲制御を応用できると考えた理由

このプロジェクトで扱ったホワイトリスト制御は、Custom GPT だけに限らない考え方です。

AI に外部情報を参照させる場合、どの情報を渡すか、どの情報を渡さないかを制御することは重要です。これは、社内ナレッジ検索、FAQ Bot、AI エージェント、業務支援ツールなど、外部情報を参照して回答する仕組みにも応用できます。

たとえば、以下のような場面でも同じ考え方が使えます。

- AI に参照させる社内文書を制限する
- 古いマニュアルや未整理メモを回答対象から外す
- プロジェクトごとに参照可能な文書を分ける
- API 経由で取得できる情報の範囲を制御する
- 許可していないファイルが AI の回答に混ざることを防ぐ

この経験から、AI 活用ではモデルや API の使い方だけでなく、AI に渡す情報の範囲設計も重要な実務的テーマになると学びました。

## このプロジェクトで学べたこと

このプロジェクトでは、AI に外部情報を参照させる場合、単に Private リポジトリの内容を読み込ませるだけでなく、どの情報を読ませるかを設計する必要があると学びました。

公開ポートフォリオは、人間が読みやすいように情報量を絞る必要があります。一方で、AI に補助させる場合は、公開ページには載せにくい細かい制作背景や判断理由も役立つことがあります。

そのため、公開資料と Private リポジトリの詳細メモを分け、Bridge API を通して許可済み Markdown だけを返す構成にしました。

また、ホワイトリスト制御は API 設計とセットで考える必要があると学びました。任意のパスを受け付ける設計ではなく、document_id を検証して許可済みパスに変換することで、参照範囲を制御しやすくなります。

この経験から、AI を使った仕組みを作る場合、API やモデルだけでなく、参照させる情報の粒度、範囲、更新しやすさ、ノイズの少なさ、取得経路の制御まで含めて設計する必要があると学びました。
