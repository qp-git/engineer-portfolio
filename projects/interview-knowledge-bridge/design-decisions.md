# Interview Knowledge Bridge 設計判断メモ

## このドキュメントの目的

このドキュメントでは、Interview Knowledge Bridge で行った設計判断を整理します。

README ではプロジェクト概要を、[architecture.md](./architecture.md) では Custom GPT、Bridge API、ホワイトリスト、GitHub API の関係を説明しています。ここでは、なぜ Private リポジトリ上の Markdown を直接読ませず中継 API にしたか、なぜホワイトリストで参照範囲を制御したか、AI に渡す情報の範囲をどのように考えたかを中心にまとめます。

## 1. PublicとPrivateで情報の役割を分けた理由

このプロジェクトでは、人間向けに公開する情報と、AI に補助情報として渡す情報を分けて考えました。

公開ポートフォリオには、プロジェクト概要、構成、設計判断、学びなど、初見の人が読みやすいように整理した情報を置きます。この公開ポートフォリオリポジトリが、その役割にあたります。

一方で、より細かいコード解説、制作背景、判断理由、作業中の補足メモなどは、公開ページには少し細かすぎる場合があります。しかし、AI に技術整理を補助させる場合には、こうした詳細情報が回答材料として役立つことがあります。

そのため、詳細な補足情報は Private リポジトリ側で管理し、必要なものだけを AI に参照させる方針にしました。

この分け方により、人間向けの公開資料は読みやすさを保ちつつ、AI には必要に応じてより細かい背景情報を渡せる構成にできます。

## 2. 中継APIとホワイトリストで参照範囲を制御した理由

Custom GPT から Private リポジトリ上の Markdown を参照する場合、リポジトリ内の情報をそのまま読ませる方法も考えられます。

しかし、Private リポジトリには、AI に渡したい情報だけでなく、作業途中のメモや古い試行錯誤も残る可能性があります。そうした情報まで AI が参照すると、回答に不要な情報が混ざったり、現在の方針と違う内容を拾ったりする可能性があります。

そこで、Custom GPT と Private GitHub リポジトリの間に Bridge API を置き、参照できる Markdown をホワイトリストで制御する構成にしました。

中継 API にすることで、以下を整理できます。

- `document_id` と GitHub 上のパスの対応を API 側で管理できる
- `project_id` によってプロジェクト単位で参照対象を絞れる
- ホワイトリストによって参照範囲を制御できる

また、ホワイトリストを機能させるには、単に許可文書の一覧を作るだけでは不十分です。

たとえば、Bridge API が任意の GitHub URL やファイルパスをそのまま受け取る設計だと、ホワイトリストに登録していないファイルを指定できる余地が残ります。その場合、ホワイトリストは一覧として存在していても、実際の参照制御としては機能しません。

そのため、Custom GPT からは任意のパスではなく `document_id` や `project_id` を受け取り、API 側で許可済みの GitHub パスに変換する設計にしました。

`document_id` を使うことで、取得できる文書を1件ずつ明示的に管理できます。さらに `project_id` を使うことで、OdaiBox、STT + ECS、Interview Knowledge Bridge など、プロジェクト単位で関連する許可済み Markdown だけを取得できます。

これにより、AI に渡すコンテキストをプロジェクト単位で絞り込めるようになり、関係ないプロジェクトの情報や古いメモが回答に混ざりにくくなります。

Custom GPT 側にも、必要に応じて `project_id` や `document_id` を指定して Bridge API を呼び出すように指示を設定します。ただし、GPT 側の指示はあくまで呼び出し方を安定させるための補助であり、参照範囲の制御そのものは API 側のホワイトリストで行う設計にしました。

## 3. ホワイトリスト制御を他のAI活用にも応用できると考えた理由

このプロジェクトで扱ったホワイトリスト制御は、Custom GPT だけに限らない考え方です。

AI に外部情報を参照させる場合、どの情報を渡すか、どの情報を渡さないかを制御することは重要です。これは他の AI エージェントなどの外部情報を参照する仕組みにも応用できます。

たとえば、以下のような場面でも同じ考え方が使えます。

- AI に参照させる社内文書を制限する
- 古いマニュアルや未整理メモを回答対象から外す
- プロジェクトごとに参照可能な文書を分ける
- API 経由で取得できる情報の範囲を制御する
- 許可していないファイルが AI の回答に混ざることを防ぐ

また、`project_id` のような単位で文書群を管理すると、AI に対して「どの領域の情報を使うか」を明示しやすくなります。これは、プロジェクト単位、部門単位、顧客単位などで参照範囲を分けたい場合にも応用できる考え方です。

この経験から、AI 活用ではモデルや API の使い方だけでなく、AI に渡す情報の範囲設計も重要な実務的テーマになると考えました。

## このプロジェクトで学べたこと

このプロジェクトでは、AI に外部情報を参照させる場合、単に Private リポジトリの内容を読み込ませるだけでなく、どの情報を読ませるかを設計する重要性を学びました。

公開ポートフォリオは、人間が読みやすいように情報量を絞る必要があります。一方で、AI に補助させる場合は、公開ページには載せにくい細かい制作背景や判断理由も、よりよい回答のために役立つことがあります。

そのため、公開資料と Private リポジトリの詳細メモを分け、Bridge API を通して許可済み Markdown だけを返す構成にしました。

また、ホワイトリスト制御は API 設計とセットで考える必要があると学びました。`document_id` や `project_id` を使って参照対象を制御することで、AI に渡す情報を必要な範囲に絞りやすくなります。

この経験から、AI を使った仕組みを作る場合、API やモデルだけでなく、参照させる情報の粒度、範囲、ノイズの少なさ、取得経路の制御まで含めて設計する必要があると学びました。
