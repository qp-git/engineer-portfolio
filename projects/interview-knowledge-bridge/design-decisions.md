# Interview Knowledge Bridge 設計判断メモ

## このドキュメントの目的

このドキュメントでは、Interview Knowledge Bridge で行った設計判断を整理します。

README ではプロジェクト概要を、[architecture.md](./architecture.md) では構成を説明しています。ここでは、なぜ中継 API にしたか、なぜホワイトリスト制御にしたか、どのように 面接練習用の情報管理を考えたかを中心にまとめます。

## 1. 人間向け資料と AI 向けコンテキストを分けた理由

ポートフォリオを整理する中で、人間に見せる資料と AI に読ませる資料では、適した情報量や粒度が違うと感じました。

人間向けの公開ポートフォリオでは、初見で読みやすいように、概要、構成、設計判断、学びを簡潔に整理する必要があります。文章が長すぎたり、細かい躓きが多すぎたりすると、読み手に負担がかかります。

一方で、面接練習用AIに回答材料として参照させる場合は、公開ページには載せにくい細かい情報も役に立ちます。たとえば、どこで詰まったか、なぜその構成にしたか、他の選択肢をどう考えたか、今ならどう改善するかといった情報です。

そのため、公開ポートフォリオと 面接練習用の詳細コンテキストを分けて管理する方針にしました。

## 2. GitHub 上のすべての文書を読ませない理由

GitHub 上に Markdown を置くだけなら、AI にすべて読ませることもできます。

しかし、面接練習や回答生成では、すべての文書が常に役立つわけではありません。関係ない情報や古いメモが混ざると、回答の焦点がずれる可能性があります。

そこで、Custom GPT が参照してよい Markdown をホワイトリストで管理することにしました。

ホワイトリスト制御により、以下を明確にできます。

- AI に読ませてよい文書
- 面接回答に使ってよい情報
- 現時点で有効な補足メモ
- プロジェクトごとの参照対象
- 公開資料と詳細コンテキストの役割分担

この設計により、公開・非公開の区別だけでなく、AI に渡す情報の範囲そのものを制御できるようにしました。

## 3. 中継APIにした理由

Custom GPT から GitHub 上の Markdown を参照する場合、直接ファイルを読ませる方法も考えられます。

しかし、直接参照にすると、どの文書を読めるのか、どの形式で返すのか、将来どのように対象文書を増やすのかを管理しづらくなります。

そこで、Custom GPT と GitHub の間に Bridge API を置く構成にしました。

中継 API にすることで、以下を整理できます。

- Custom GPT が呼び出すインターフェースを固定できる
- 文書 ID と GitHub 上のパスの対応を API 側で管理できる
- ホワイトリストによって参照範囲を制御できる
- 将来的に保存先や文書構成が変わっても、GPT 側の呼び出し方を大きく変えずに済む
- 公開資料と補足メモを同じ設計思想で扱える

このように、中継 API は単なる取得処理ではなく、AI に渡す情報の入口を管理する役割を持っています。

## 4. Markdownを参照対象にした理由

参照対象は Markdown としました。

Markdown は、人間が直接読んでも分かりやすく、GitHub 上でも管理しやすい形式です。また、見出しや箇条書きで情報を整理しやすいため、AI に渡すコンテキストとしても扱いやすい形式です。

特に面接準備では、プロジェクトごとに以下のような情報を残しやすくなります。

- 概要
- 構成
- 設計判断
- 詰まった点
- 改善案
- 面接で聞かれそうな質問
- 回答の材料

Markdown にすることで、人間が編集しやすく、AI も参照しやすい情報管理ができます。

## 5. ホワイトリストをセキュリティだけでなく情報設計として扱った理由

このプロジェクトでは、ホワイトリストを単なるアクセス制限としてではなく、AI に渡す情報を設計する仕組みとして扱いました。

AI に大量の情報を渡せばよいわけではありません。情報が多すぎると、必要な論点が埋もれたり、回答に関係ない内容が混ざったりします。

そのため、面接練習で使う情報は、以下の観点で選ぶ必要があります。

- 回答に使ってよいか
- 現在の自己PRやポートフォリオ方針と合っているか
- 深掘り質問に答える材料になるか
- 古い判断や未整理のメモではないか
- 公開資料には重すぎるが、AI には読ませたい補足か

ホワイトリストは、このような情報選別を API の仕組みとして反映するためのものです。

## このプロジェクトで学べたこと

このプロジェクトでは、AI 活用において、モデルに何を読ませるかも重要な設計対象になると学びました。

公開ポートフォリオは、人間にとって読みやすい情報量に絞る必要があります。一方で、面接練習用AIに参照させる情報は、細かい躓きや判断理由まで含んでいた方が、深掘り質問に対応しやすくなります。

そのため、公開資料と AI 用詳細コンテキストを分け、Bridge API を通して許可済み Markdown だけを返す構成にしました。

この経験から、AI を使った仕組みを作る場合、API やモデルだけでなく、参照させる情報の粒度、範囲、更新しやすさ、ノイズの少なさまで含めて設計する必要があると学びました。
