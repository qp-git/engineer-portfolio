# Interview Knowledge Bridge

## 概要

Interview Knowledge Bridge は、Custom GPT から Private GitHub リポジトリ上の許可済み Markdown を参照できるようにするための中継 API です。

公開ポートフォリオには、人間向けに読みやすく整理したプロジェクト概要や設計判断メモを置きます。一方で、より細かいコード解説、制作背景、判断理由、作業中の補足メモなどは、Private リポジトリ側で管理します。

この API では、Custom GPT が Private リポジトリ内のすべての情報を参照するのではなく、ホワイトリストで許可した Markdown だけを取得できるようにしました。これにより、AI に必要な補足情報を渡しつつ、参照範囲を明確に制御できる構成にしています。

この構成を通して、Custom GPT Actions、API Gateway、Lambda、GitHub API の連携だけでなく、AI に外部情報を参照させる際の情報管理やアクセス範囲の設計についても学びました。

## 作成背景

ポートフォリオを整理していく中で、人間に見せる公開資料と、AI に参照させたい詳細な補足情報は役割が異なると感じました。

公開資料では、初見の人が読みやすいように、概要、構成、設計判断、学びを簡潔にまとめる必要があります。一方で、AI に技術整理を補助させる場合は、実際に詰まった点、なぜその構成にしたか、他の選択肢をどう考えたか、今ならどう改善するかといった細かい情報も役立ちます。

そこで、公開ポートフォリオとは別に、Private リポジトリ側で詳細な補足メモを管理し、Custom GPT から必要な文書だけ取得できる中継 API を作成しました。

## 目的

このプロジェクトの目的は、Custom GPT に対して、Private リポジトリ内の許可済み Markdown だけを明示的に渡すことです。

主な目的は以下です。

- 人間向けの公開ポートフォリオと、AI に渡す詳細コンテキストを分ける
- Custom GPT が参照できる Markdown をホワイトリストで制御する
- Private リポジトリ内のすべてを AI に読ませない
- 回答生成や技術整理に使ってよい補足情報だけを API 経由で返す
- 将来的に文書が増えても、参照範囲を同じ仕組みで管理できるようにする

## 構成の概要

基本的な流れは以下です。

    Custom GPT
      ↓ Actions 経由でリクエスト
    API Gateway
      ↓
    Lambda
      ↓ document_id を検証
    Whitelist
      ↓ 許可済みの場合のみ取得
    GitHub API
      ↓ Markdown を返却
    Custom GPT

Bridge API は、Custom GPT からのリクエストを受け取り、ホワイトリストに登録された Markdown だけを取得して返します。

重要なのは、任意の GitHub パスや URL をそのまま受け取るのではなく、API 側で管理している document_id をもとに対象ファイルを決める点です。中継 API の設計が甘いと、ホワイトリストを用意していても参照制御が効かなくなるため、入口の設計も重要な検証ポイントになりました。

## 使用技術・要素

- Custom GPT Actions
- OpenAPI schema
- Amazon API Gateway
- AWS Lambda
- GitHub API
- Private GitHub リポジトリ
- Markdown
- ホワイトリストによる参照制御

## 実装・検証したこと

- Custom GPT から外部 API を呼び出す構成を検証
- API Gateway と Lambda を使った中継 API を作成
- GitHub API 経由で Private リポジトリ上の Markdown を取得
- 参照可能な Markdown をホワイトリストで制御
- document_id と GitHub 上のパスの対応を API 側で管理
- 任意パスを直接指定させず、許可済み文書だけを返す構成を検討

## このプロジェクトで学んだこと

このプロジェクトでは、AI に外部情報を参照させる場合、単に Private リポジトリを用意するだけでなく、どの情報を読ませるかを設計する必要があると学びました。

特に、Private リポジトリには詳細な補足情報を残せる一方で、そのすべてが AI の回答材料として適しているとは限りません。古いメモや未整理の作業ログまで参照対象にすると、回答の焦点がずれる可能性があります。

そのため、AI が参照する文書をホワイトリストで管理し、必要な情報だけを API 経由で渡す構成にしました。

また、ホワイトリストは一覧を作るだけでは不十分で、API が任意のパスや URL を受け付けてしまうと参照制御として機能しません。document_id を API 側で検証し、許可済みのパスにだけ変換する設計が重要だと学びました。

この経験から、AI 活用では、モデルそのものだけでなく、参照させる情報の粒度、範囲、取得経路、アクセス制御も重要な設計対象になると学びました。
