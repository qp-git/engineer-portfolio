# Alexa Skill 構成メモ

このプロジェクトは、Alexa スキルを題材に、音声UI、AWS Lambda、外部API連携を学習したプロジェクトです。

当初は、Alexa に話しかけるだけで、2地点の天気や通勤に関係する情報をまとめて確認できる仕組みを目指しました。

## 全体構成

Alexa Skill は、ユーザーの音声入力を Alexa 側で解釈し、その結果を Lambda に渡して処理する構成です。

    ユーザー
      ↓ 音声入力
    Alexa
      ↓ インテント判定
    Alexa Skills Kit
      ↓ リクエスト
    AWS Lambda
      ↓ 外部API呼び出し
    天気API / 遅延情報API
      ↓ 結果
    AWS Lambda
      ↓ レスポンス
    Alexa
      ↓ 読み上げ
    ユーザー

この構成により、ユーザーは画面を開かず、音声だけで必要な情報を取得できます。

## API Gatewayを使わない構成

このプロジェクトでは、Alexa Skills Kit から Lambda 関数を直接呼び出す構成にしています。

通常の Web アプリケーションでは、外部から Lambda を呼び出すために API Gateway を前段に置くことがあります。

しかし Alexa Skill では、Alexa Skills Kit のエンドポイント設定で Lambda 関数の ARN を指定できます。そのため、Alexa から Lambda を呼び出すためだけに API Gateway を用意する必要はありません。

    Alexa Skills Kit
      ↓ Lambda ARNをエンドポイントとして指定
    AWS Lambda

この構成により、API Gateway を挟まずに、Alexa のリクエストを Lambda で処理できます。

当初は「外部サービスから Lambda を呼ぶなら API Gateway が必要」と考えていましたが、Alexa Skill ではサービス側に Lambda 連携の仕組みが用意されているため、よりシンプルな構成にできると分かりました。

この学びから、AWSサービスを組み合わせる際には、一般的なWeb API構成をそのまま当てはめるのではなく、対象サービスが持っている連携方法を確認することが重要だと感じました。

## 使用技術

| 分類 | 技術・サービス | 役割 |
|---|---|---|
| 音声UI | Alexa | ユーザーの音声入力・読み上げ |
| スキル開発 | Alexa Skills Kit | スキル設定、インテント定義、テスト |
| バックエンド | AWS Lambda | Alexa からのリクエスト処理 |
| 外部API | 天気API | 2地点の天気情報取得 |
| 外部API調査 | 遅延情報API | 電車遅延情報取得の検証 |
| 実装言語 | Node.js / JavaScript | Lambda 関数実装 |
| 設定情報 | 環境変数など | APIキーや地点情報の管理候補 |

## Alexa Skill の処理の流れ

Alexa Skill では、ユーザーが発話した内容を Alexa が解釈し、対応するインテントとして Lambda に渡します。

    1. ユーザーが Alexa に話しかける
    2. Alexa がスキルを起動する
    3. Alexa Skills Kit が発話内容を解析する
    4. 対応するインテントを判定する
    5. Lambda にリクエストを送る
    6. Lambda が必要なAPIを呼び出す
    7. Lambda が読み上げ用の文を組み立てる
    8. Alexa がユーザーへ読み上げる

この流れにより、音声入力とバックエンド処理を分けて考えられます。

## Invocation Name

Alexa Skill では、ユーザーがスキルを呼び出すための名前を Invocation Name として設定します。

たとえば、ユーザーが「アレクサ、〇〇を開いて」と話しかけると、Alexa はその Invocation Name に対応するスキルを起動します。

この設定は、単なる名前ではなく、ユーザーが自然に呼び出せるかどうかに関わります。

音声UIでは、画面上のボタンやURLがないため、呼び出しやすい名前を考えることも設計の一部になります。

## インテントと Lambda

Alexa Skills Kit 側では、ユーザーが何をしたいのかをインテントとして定義します。

このプロジェクトでは、天気を聞く、通勤情報を確認するといった用途を想定しました。

    ユーザーの発話
      ↓
    インテント
      ↓
    Lambda の処理分岐

Lambda 側では、受け取ったインテントに応じて処理を分岐します。

たとえば、天気確認のインテントであれば天気APIを呼び出し、結果を読み上げ用の文章に整形します。

## 2地点の天気取得

このプロジェクトでは、1地点だけでなく、2地点の天気情報を取得する構成を検証しました。

    Lambda
      ↓
    地点Aの天気API呼び出し
      ↓
    地点Bの天気API呼び出し
      ↓
    結果をまとめる
      ↓
    Alexa向け読み上げ文を生成

2地点を扱うことで、単純なAPI呼び出しだけでなく、複数の外部API結果をまとめて文章化する流れを確認できます。

音声で読み上げる場合、画面表示とは違い、情報量が多すぎると聞き取りにくくなります。

そのため、必要な情報を短く整理して返すことが重要になります。

## 遅延情報APIの調査

天気情報に加えて、電車の遅延情報も取得する構成を検討しました。

ただし、遅延情報APIは利用条件や提供範囲に制約があり、無料利用や個人利用で扱える情報が限られる場合があります。

この調査では、以下のような点を確認しました。

- どのAPIで運行情報を取得できるか
- 無料プランで利用できる範囲
- 法人契約や有料契約が必要になる条件
- APIキーの取得条件
- レスポンス形式
- 利用規約上の制限

結果として、遅延情報の実装は天気情報よりも利用条件の確認が重要だと分かりました。

## 外部API呼び出しと失敗時の扱い

複数の外部APIを呼び出す場合、一部のAPIだけが失敗する可能性があります。

たとえば、天気情報は取得できても、遅延情報APIがタイムアウトするケースがあります。

    Lambda
      ↓
    天気API 成功
      ↓
    遅延情報API 失敗
      ↓
    どこまで読み上げるか判断

このような場合、全体を失敗にするのか、取得できた情報だけを返すのかを考える必要があります。

音声UIでは、エラーをそのまま返すと利用者に分かりにくいため、自然な読み上げ文に変換することも重要です。

## Lambda のタイムアウト

Alexa Skill では、ユーザーが発話してから応答までの時間が長すぎると使いにくくなります。

また、Lambda にはタイムアウト設定があります。

外部API呼び出しが遅い場合、Lambda 側でタイムアウトしたり、Alexa 側の応答待ち時間に間に合わない可能性があります。

そのため、以下の観点が重要になります。

- API呼び出しのタイムアウトを設定する
- 複数APIを順番に呼ぶか並列に呼ぶか考える
- 一部失敗時に取得済み情報だけ返すか考える
- 読み上げ文を短くする
- 応答速度を優先する

この点は、通常のWebアプリ以上に音声UIでは重要だと感じました。

## 音声UIとしての注意点

Alexa Skill では、画面に情報を一覧表示するわけではありません。

そのため、読み上げ文の設計が重要です。

意識した点は以下です。

- 長すぎる文章を避ける
- 重要な情報から読む
- 聞き返されにくい表現にする
- エラー時も自然な文章にする
- 数字や地名を聞き取りやすくする
- 一度に詰め込みすぎない

音声UIでは、ユーザーが読み返せない前提で、情報量と分かりやすさのバランスを取る必要があります。

## 学習上のポイント

このプロジェクトでは、Alexa Skill を通して以下を学びました。

- 音声UIの基本構造
- Alexa Skills Kit と Lambda の連携
- インテント設計
- Invocation Name の役割
- Lambda での外部API呼び出し
- 複数APIの結果をまとめる処理
- 外部APIの利用条件調査
- タイムアウトや部分失敗への対応
- 音声で返す文章の設計
