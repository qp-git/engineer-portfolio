# Alexa Skill 設計判断メモ

このプロジェクトでは、Alexa Skill を題材に、音声 UI、AWS Lambda、外部 API 連携を学習しました。

ポートフォリオとしては、以下の 5 点を中心に整理します。

1. Alexa Skill から Lambda を直接呼び出す構成
2. 音声 UI 向けのレスポンス設計
3. 最小構成からの段階的な検証
4. 複数 API 連携時の部分失敗設計
5. 外部 API の利用条件まで含めた切り分け

## 1. Alexa Skill から Lambda を直接呼び出す構成

当初は、Alexa から Lambda を呼び出すために API Gateway が必要だと考えていました。

しかし Alexa Skill では、Alexa Skills Kit のエンドポイント設定で Lambda 関数の ARN を直接指定できます。そのため、この構成では API Gateway を挟まず、Alexa Skills Kit から Lambda を直接呼び出す形にしました。

この経験により、AWS のサービスを組み合わせる際には、一般的な Web API 構成をそのまま当てはめるのではなく、サービスごとの連携方法を確認することが重要だと学びました。

また、Alexa Skill は Lambda の入口と出口の両方を担えます。ユーザーの音声入力をきっかけに Lambda を起動し、Lambda の処理結果を Alexa に返すことで、ユーザーへ音声で応答できます。

## 2. 音声 UI 向けのレスポンス設計

Alexa Skill では、API から取得した情報をそのまま返せばよいわけではありません。

API レスポンスは機械向けのデータであり、ユーザーが音声で聞いて理解しやすい形に整える必要があります。

意識した点は以下です。

- 長すぎる文章を避ける
- 重要な情報から伝える
- 数字や地名を聞き取りやすくする
- エラー時も自然な文章にする
- 画面がなくても内容が分かるようにする

音声 UI では、ユーザーが情報を見返せません。そのため、読み上げ文は短く、自然で、必要な情報に絞ることが重要だと学びました。

## 3. 最小構成からの段階的な検証

最初から機能を増やしすぎず、まずは固定文を返す Lambda を作り、その後で天気 API 連携、Alexa Skill 連携へ進めました。

この順番にしたことで、問題が起きたときに原因を切り分けやすくなりました。

確認した流れは以下です。

- Lambda 単体で固定文を返せるか
- Lambda から外部 API を呼び出せるか
- API の取得結果を読み上げ文に整形できるか
- Alexa Skill から Lambda を呼び出せるか
- Alexa が Lambda の戻り値を読み上げられるか

このように段階を分けることで、Alexa 側の設定問題、Lambda 側の実装問題、外部 API の問題を分けて確認できました。

## 4. 複数 API 連携時の部分失敗設計

2 地点の天気や遅延情報など、複数の外部 API を呼び出す構成では、一部の API だけが失敗する可能性があります。

たとえば、天気情報は取得できても、遅延情報だけ取得できないケースがあります。

このとき、全体をエラーにするのか、取得できた情報だけを返すのかを考える必要があります。

朝レポートでは、天気情報を本体、遅延情報を追加情報として扱う方針にしました。利用者としては、遅延情報が取得できなくても、天気だけでも返ってきた方が価値があるためです。

また、`Promise.all()` はどれか 1 つの Promise が失敗すると全体が失敗として扱われます。そのため、一部成功でも価値がある処理では、個別の `try / catch` や `Promise.allSettled()` の利用も検討対象になると学びました。

## 5. 外部 API の利用条件まで含めた切り分け

当初は、天気情報に加えて電車の遅延情報も Alexa で読み上げる構成を想定していました。

しかし、遅延情報 API は天気情報よりも利用条件の確認が難しく、無料プランや個人利用では取得できる情報に制限がありました。

そのため、無理に完成扱いにせず、API 調査と制約確認までを学習成果として整理しました。

この判断により、以下を学べました。

- 作りたい機能があっても、API 利用条件によって実装可否が変わる
- 無料プランで検証できる範囲には限界がある
- 法人契約や有料契約が必要な API もある
- 技術的に呼び出せることと、規約上利用できることは別
- 未実装の理由を説明できる形で残すことも重要

学習プロジェクトでは、動いた部分だけでなく、調査して断念した理由も残すことで、実務に近い判断材料になります。

## 今後の改善余地

今後改善するなら、以下を検討します。

- 天気情報の読み上げ文をさらに自然にする
- API 失敗時の返答パターンを増やす
- `Promise.allSettled()` などを使った部分成功対応
- API ごとのタイムアウト設定
- Invocation Name の改善
- API キーの Secrets Manager 管理
- 遅延情報 API の利用条件を再調査
- ログ出力とエラー調査手順の整理
