# Alexa Skill 設計判断メモ

## このドキュメントの目的

このドキュメントでは、Alexa Skill プロジェクトで行った設計判断を整理します。

README ではプロジェクト概要を、architecture.md では Alexa Skill、Lambda、外部 API の関係を説明しています。

ここでは、なぜ Alexa Skill を題材にしたか、なぜ Lambda を使ったか、音声 UI と外部 API 連携でどのような点を意識したかを中心にまとめます。

## 1. Alexa Skill を題材にした理由

Alexa Skill を題材にした理由は、画面を使わない UI と AWS Lambda の連携を学びやすいからです。

通常の Web アプリでは、ユーザーはボタンやフォームを見て操作できます。一方、Alexa Skill では、ユーザーは音声でスキルを呼び出し、結果も音声で受け取ります。

そのため、機能そのものだけでなく、呼び出しやすさ、聞き取りやすさ、返答の長さも設計対象になります。

このプロジェクトでは、音声 UI を通して Lambda を起動し、外部 API の結果を読み上げ文として返す一連の流れを確認することを目的にしました。

## 2. Lambda をバックエンドにした理由

Alexa Skill のバックエンドには AWS Lambda を利用しました。

Lambda を使うことで、Alexa からのリクエストを受け取り、外部 API を呼び出して、読み上げ用のレスポンスを返す構成を小さく作れます。

Alexa Skill は、常時起動しているサーバーに接続し続けるというより、ユーザーが呼び出したときに処理できればよい用途です。そのため、イベント駆動で実行できる Lambda と相性がよいと判断しました。

また、小規模な検証では常時起動サーバーを用意せずに始められるため、音声 UI と外部 API 連携の実装に集中しやすくなります。

## 3. API Gateway を使わなかった理由

当初は、外部から Lambda を呼び出すには API Gateway が必要だと考えていました。

しかし Alexa Skill では、Alexa Skills Kit のエンドポイントとして Lambda 関数の ARN を直接指定できます。そのため、この構成では API Gateway を挟まず、Alexa Skills Kit から Lambda を直接呼び出す形にしました。

この経験により、AWS のサービスを組み合わせる際には、一般的な Web API 構成をそのまま当てはめるのではなく、サービスごとの連携方法を確認することが重要だと学びました。

Alexa Skill は、ユーザーの音声入力を Lambda の入口にし、Lambda の処理結果を音声で返す出口にもなります。この点が、通常の Web API とは異なる大きな学びでした。

## 4. 最初に 2 地点の天気取得を実装した理由

最初の実装対象として、自宅と勤務地の 2 地点の天気取得を選びました。

天気情報は API の用途が分かりやすく、結果も音声で返しやすいため、Alexa Skill と Lambda の連携を確認する題材として適しています。

また、1 地点だけではなく 2 地点を扱うことで、複数の外部 API 呼び出し、取得結果の結合、読み上げ文への整形、情報量の調整を確認できます。

音声 UI では、情報を画面に並べることができません。そのため、2 地点分の情報であっても、聞き取りやすい長さにまとめる必要があります。

## 5. Invocation Name と読み上げ文を意識した理由

Alexa Skill では、ユーザーがスキルを呼び出すための名前を Invocation Name として設定します。

Web アプリであれば、URL やボタン、メニューから機能へアクセスできます。しかし音声 UI では、ユーザーが言葉でスキルを呼び出す必要があります。

そのため、Invocation Name は単なる名前ではなく、ユーザーが自然に発話できるかどうかに関わる設計要素です。

また、API から取得した情報をそのまま返せばよいわけではありません。API レスポンスは機械向けのデータであり、ユーザーが音声で聞いて理解しやすい文章に整える必要があります。

このプロジェクトでは、長すぎる文章を避けること、重要な情報から伝えること、数字や地名を聞き取りやすくすることを意識しました。

## 6. 外部 API の制約を確認した理由

Alexa Skill 単体では、固定の文章を読み上げるだけでも動作します。しかし実用的なスキルにするには、外部 API から現在の情報を取得する必要があります。

このプロジェクトでは、天気 API や遅延情報 API の調査を通して、API キーの取得方法、無料プランで利用できる範囲、レスポンス形式、利用規約、商用利用や法人契約の条件などを確認しました。

特に遅延情報 API では、利用条件や提供範囲に制約があり、無料・個人利用では目的の情報を扱いにくい場合があると分かりました。

そのため、無理に完成扱いにせず、API 調査と制約確認までを学習成果として整理しました。

この経験から、外部 API を使うプロジェクトでは、技術的に呼び出せるかだけでなく、利用条件や提供範囲まで確認することが重要だと学びました。

## 7. 複数 API 呼び出しと失敗時の返し方を意識した理由

2 地点の天気や遅延情報など、複数の外部 API を呼び出す構成では、一部の API だけが失敗する可能性があります。

たとえば、片方の天気情報は取得できても、もう片方の情報取得がタイムアウトすることがあります。また、天気情報は取得できても、遅延情報だけ取得できないケースもあります。

このとき、全体をエラーにするのか、取得できた情報だけを返すのかを考える必要があります。

特に音声 UI では、単にエラー文を返すだけでは利用者に分かりにくくなります。そのため、取得できた情報は返す、取得できなかった情報は自然な文章で伝える、失敗した API 名をそのまま読ませない、といった返し方を考えました。

また、複数 API を並列に呼び出す場合、Promise.all はどれか 1 つの Promise が失敗すると全体が失敗として扱われます。この挙動により、一部の API だけ失敗した場合でも、取得できていた情報まで返せなくなる可能性があります。

この経験から、複数 API を扱う場合は、成功時だけでなく、部分失敗時の扱いも設計対象になると学びました。

## 8. 実機検証で学んだこと

Alexa Skill の検証では、テキストシミュレータでは動く発話が、実機や定型アクションではうまく認識されないことがありました。

また、Fire TV Stick などの関連デバイスがあると、定型アクションのカスタム文が意図しない向き先に流れることもありました。

この経験から、音声 UI ではコードだけでなく、実際の利用環境によって挙動が変わる可能性があると分かりました。

Web アプリでは URL にアクセスすれば対象が明確ですが、音声 UI では同じアカウントや同じ空間に複数端末がある場合、どの端末が反応するかも利用体験に影響します。

## このプロジェクトで学べたこと

このプロジェクトを通して、Alexa Skill と Lambda の基本構成だけでなく、音声 UI 特有の設計や外部 API 利用時の制約を学べました。

特に、Alexa Skill が Lambda の入口と出口の両方を担えること、API Gateway を使わずに Lambda と連携できること、Invocation Name や読み上げ文も設計対象になることが大きな学びでした。

また、外部 API 連携では、実装前に利用条件を確認すること、一部失敗時の返し方を考えること、技術的に作れることと規約上利用できることを分けて考えることが重要だと分かりました。

学習プロジェクトとしては、完成した機能だけでなく、外部 API 制約や音声 UI 特有の難しさを含めて整理することを重視しました。
