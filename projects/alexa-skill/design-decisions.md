# Alexa Skill 設計判断メモ

このプロジェクトでは、Alexa Skill を題材に、音声 UI、AWS Lambda、外部 API 連携を学習しました。

ここでは、実装・通信の流れに沿って、主な設計判断を整理します。

## 1. Alexa Skill を音声入力の入口にした理由

Alexa Skill を題材にした理由は、画面を使わない UI と AWS Lambda の連携を学びやすいからです。

通常の Web アプリでは、ユーザーはボタンやフォームを見て操作できます。一方、Alexa Skill では、ユーザーは音声でスキルを呼び出し、結果も音声で受け取ります。

そのため、以下のような観点を学べます。

* 音声だけで機能を呼び出す設計
* Invocation Name の重要性
* Lambda でのリクエスト処理
* 外部 API の結果を読み上げ用に整形する処理
* 長すぎない応答文の設計
* 応答速度やタイムアウトへの意識

音声 UI では、機能そのものだけでなく、呼び出しやすさも設計要素になると学びました。

## 2. Alexa Skill から Lambda を直接呼び出す構成にした理由

当初は、Alexa から Lambda を呼び出すために API Gateway が必要だと考えていました。

しかし Alexa Skill では、Alexa Skills Kit のエンドポイント設定で Lambda 関数の ARN を直接指定できます。そのため、この構成では API Gateway を挟まず、Alexa Skills Kit から Lambda を直接呼び出す形にしました。

この経験により、AWS のサービスを組み合わせる際には、一般的な Web API 構成をそのまま当てはめるのではなく、サービスごとの連携方法を確認することが重要だと学びました。

また、Alexa Skill は Lambda の入口と出口の両方を担えます。ユーザーの音声入力をきっかけに Lambda を起動し、Lambda の処理結果を Alexa に返すことで、ユーザーへ音声で応答できます。

## 3. Lambda から外部 API を呼び出す構成にした理由

Alexa Skill 単体では、固定の文章を読み上げるだけでも動作します。

しかし実用的なスキルにするには、外部 API から現在の情報を取得する必要があります。

このプロジェクトでは、まず 2 地点の天気取得を実装対象にしました。天気情報は API の用途が分かりやすく、結果も音声で返しやすいため、Alexa Skill と Lambda の連携を確認する題材として適していると考えました。

また、1 地点だけではなく 2 地点を扱うことで、以下を確認できます。

* 複数の外部 API 呼び出し
* 取得結果の結合
* 読み上げ文への整形
* 情報量の調整
* API 失敗時の扱い

最初から機能を増やしすぎず、固定文の返却、Lambda 単体テスト、外部 API 連携、Alexa 連携の順で進めることで、問題の切り分けをしやすくしました。

## 4. 取得結果を音声向けに整形した理由

Alexa Skill では、API から取得した情報をそのまま返せばよいわけではありません。

API レスポンスは機械向けのデータであり、ユーザーが音声で聞いて理解しやすい形に整える必要があります。

意識した点は以下です。

* 長すぎる文章を避ける
* 重要な情報から伝える
* 数字や地名を聞き取りやすくする
* エラー時も自然な文章にする
* 画面がなくても内容が分かるようにする

音声 UI では、ユーザーが情報を見返せません。そのため、読み上げ文は短く、自然で、必要な情報に絞ることが重要だと学びました。

## 5. 遅延情報 API の追加検証を調査結果として整理した理由

当初は、天気情報に加えて電車の遅延情報も Alexa で読み上げる構成を想定していました。

しかし、遅延情報 API は天気情報よりも利用条件の確認が難しく、無料プランや個人利用では取得できる情報に制限がありました。

そのため、無理に完成扱いにせず、API 調査と制約確認までを学習成果として整理しました。

この判断により、以下を学べました。

* 作りたい機能があっても、API 利用条件によって実装可否が変わる
* 無料プランで検証できる範囲には限界がある
* 法人契約や有料契約が必要な API もある
* 技術的に呼び出せることと、規約上利用できることは別
* 未実装の理由を説明できる形で残すことも重要

また、複数の外部 API を呼び出す構成では、一部の API だけが失敗する可能性があります。

朝レポートでは、天気情報を本体、遅延情報を追加情報として扱う方針にしました。利用者としては、遅延情報が取得できなくても、天気だけでも返ってきた方が価値があるためです。

`Promise.all()` はどれか 1 つの Promise が失敗すると全体が失敗として扱われます。そのため、一部成功でも価値がある処理では、個別の `try / catch` や `Promise.allSettled()` の利用も検討対象になると学びました。

## 今後の改善余地

今後改善するなら、以下を検討します。

* 天気情報の読み上げ文をさらに自然にする
* API 失敗時の返答パターンを増やす
* `Promise.allSettled()` などを使った部分成功対応
* API ごとのタイムアウト設定
* Invocation Name の改善
* API キーの Secrets Manager 管理
* 遅延情報 API の利用条件を再調査
* ログ出力とエラー調査手順の整理
