# Alexa Skill 設計判断メモ

このプロジェクトでは、Alexa Skill を題材に、音声UI、AWS Lambda、外部API連携を学習しました。

単に Alexa で天気を読み上げるだけでなく、音声UIではどのように情報を返すべきか、外部APIの制約をどう調査するか、API失敗時にどう扱うかを意識しました。

## Alexa Skill を題材にした理由

Alexa Skill を題材にした理由は、画面を使わないUIとAWS Lambdaの連携を学びやすいからです。

通常のWebアプリでは、ユーザーはボタンやフォームを見て操作できます。

一方、Alexa Skill では、ユーザーは音声でスキルを呼び出し、結果も音声で受け取ります。

そのため、以下のような観点を学べます。

- 音声だけで機能を呼び出す設計
- Invocation Name の重要性
- インテント設計
- Lambda でのリクエスト処理
- 外部APIの結果を読み上げ用に整形する処理
- 長すぎない応答文の設計
- 応答速度やタイムアウトへの意識

Web画面がない分、ユーザーが迷わず使える呼び出し方や、聞き取りやすい返答を考える必要があると感じました。

## 最初に2地点の天気取得を実装した理由

最初の実装対象として、2地点の天気取得を選びました。

天気情報はAPIの用途が分かりやすく、結果も音声で返しやすいため、Alexa Skill と Lambda の連携を確認する題材として適しています。

また、1地点だけではなく2地点を扱うことで、以下を確認できます。

- 複数の外部API呼び出し
- 取得結果の結合
- 読み上げ文への整形
- 情報量の調整
- API失敗時の扱い

音声UIでは、情報を画面に並べることができません。

そのため、2地点分の情報であっても、聞き取りやすい長さにまとめる必要があります。

## Lambda をバックエンドにした理由

Alexa Skill のバックエンドには AWS Lambda を利用しました。

Lambda を使うことで、Alexa からのリクエストを受け取り、外部APIを呼び出して、読み上げ用のレスポンスを返す構成を小さく作れます。

Lambda を選んだ理由は以下です。

- Alexa Skills Kit と連携しやすい
- 常時起動サーバーが不要
- スキル実行時だけ処理できる
- 小規模な検証では低コストで始めやすい
- 外部API呼び出しの実装に集中できる

Alexa Skill は、常にサーバーへ接続し続けるというより、ユーザーが呼び出したときに処理できればよい用途です。

そのため、Lambda のイベント駆動構成と相性がよいと判断しました。

## API Gatewayを使わなかった理由

当初は、Alexa から Lambda を呼び出すために API Gateway が必要だと考えていました。

しかし Alexa Skill では、Alexa Skills Kit のエンドポイント設定で Lambda 関数の ARN を直接指定できます。そのため、この構成では API Gateway を挟まず、Alexa Skills Kit から Lambda を直接呼び出す形にしました。

この経験により、AWS のサービスを組み合わせる際には、一般的な Web API 構成をそのまま当てはめるのではなく、サービスごとの連携方法を確認することが重要だと学びました。

また、「外部から Lambda を呼ぶなら API Gateway が必要」と考えてしまいがちですが、Alexa Skill のようにサービス側が Lambda 連携の仕組みを持っている場合は、構成がよりシンプルになることがあります。

## Invocation Name を意識した理由

Alexa Skill では、ユーザーがスキルを呼び出すための名前を Invocation Name として設定します。

Webアプリであれば、URLやボタン、メニューから機能へアクセスできます。

しかし音声UIでは、ユーザーが言葉でスキルを呼び出す必要があります。

そのため、Invocation Name は単なる名前ではなく、ユーザーが自然に発話できるかどうかに関わる設計要素です。

このプロジェクトでは、音声UIでは機能そのものだけでなく、呼び出しやすさも重要だと学びました。

## 外部API連携を重視した理由

Alexa Skill 単体では、固定の文章を読み上げるだけでも動作します。

しかし実用的なスキルにするには、外部APIから現在の情報を取得する必要があります。

このプロジェクトでは、天気APIや遅延情報APIの調査を通して、以下を確認しました。

- APIキーの取得方法
- 無料プランで利用できる範囲
- レスポンス形式
- 利用規約
- 商用利用や法人契約の条件
- タイムアウトや失敗時の挙動
- APIの情報更新頻度

特に遅延情報APIでは、利用条件や提供範囲に制約があり、無料・個人利用では目的の情報を扱いにくい場合があると分かりました。

この経験から、外部APIを使うプロジェクトでは、実装前に利用条件を確認することが重要だと学びました。

## 遅延情報APIの実装を完了扱いにしなかった理由

当初は、天気情報に加えて電車の遅延情報も Alexa で読み上げる構成を想定していました。

しかし、遅延情報APIは天気情報よりも利用条件の確認が難しく、無料プランや個人利用では取得できる情報に制限がありました。

そのため、無理に完成扱いにせず、API調査と制約確認までを学習成果として整理しました。

この判断により、以下を学べました。

- 作りたい機能があっても、API利用条件によって実装可否が変わる
- 無料プランで検証できる範囲には限界がある
- 法人契約や有料契約が必要なAPIもある
- 技術的に呼び出せることと、規約上利用できることは別
- 未実装の理由を説明できる形で残すことも重要

学習プロジェクトでは、動いた部分だけでなく、調査して断念した理由も残すことで、実務に近い判断材料になります。

## 複数API呼び出しで部分失敗を意識した理由

2地点の天気や遅延情報など、複数の外部APIを呼び出す構成では、一部のAPIだけが失敗する可能性があります。

たとえば、片方の天気APIは成功しても、もう片方のAPIがタイムアウトすることがあります。

また、天気情報は取得できても、遅延情報だけ取得できないケースもあります。

このとき、全体をエラーにするのか、取得できた情報だけを返すのかを考える必要があります。

音声UIでは、単にエラー文を返すだけでは利用者に分かりにくくなります。

そのため、以下のような設計が必要だと考えました。

- 取得できた情報は返す
- 取得できなかった情報は自然な文章で伝える
- 失敗したAPI名をそのまま読ませない
- ユーザーが次にどうすればよいか分かる文にする
- 全体失敗と部分失敗を分ける

この経験により、複数APIを扱う場合は、成功時だけでなく失敗時の返し方も設計対象になると学びました。

## Lambda のタイムアウトを意識した理由

Alexa Skill では、ユーザーが話しかけてから応答が返るまでの時間が重要です。

外部APIの応答が遅い場合、LambdaのタイムアウトやAlexa側の応答待ち時間に影響します。

このプロジェクトでは、外部API呼び出しが増えるほど、応答速度とタイムアウト設計が重要になると分かりました。

意識した点は以下です。

- Lambdaのタイムアウト設定
- 外部API呼び出し側のタイムアウト
- 複数APIを並列に呼ぶか順番に呼ぶか
- 一部失敗時に処理を続けるか
- 応答文を短くして読み上げ時間を抑える
- ユーザーを待たせすぎない設計

音声UIでは、画面上のローディング表示が使えないため、応答の遅さがそのまま使いにくさにつながります。

## Promise.all の扱いで学んだこと

複数のAPIを並列に呼び出すとき、Promise.all を使うと処理をまとめやすくなります。

一方で、Promise.all はどれか1つのPromiseが失敗すると、全体が失敗として扱われます。

この挙動により、片方のAPIだけ失敗した場合でも、取得できていた情報まで返せなくなる可能性があります。

この経験から、複数APIを扱う場合は、以下を考える必要があると学びました。

- 全部成功しないと意味がない処理なのか
- 一部成功でもユーザーに価値がある処理なのか
- Promise.all でまとめてよいのか
- 個別に try / catch すべきか
- Promise.allSettled のような扱いを検討すべきか

Alexa Skill のように利用者へ情報を返す用途では、一部失敗でも取得済み情報を返せる設計の方が自然な場合があります。

## 読み上げ文の設計を重視した理由

Alexa Skill では、APIから取得した情報をそのまま返せばよいわけではありません。

APIレスポンスは機械向けのデータであり、ユーザーが音声で聞いて理解しやすい形に整える必要があります。

意識した点は以下です。

- 長すぎる文章を避ける
- 重要な情報から伝える
- 数字や地名を聞き取りやすくする
- エラー時も自然な文章にする
- 同じ表現を繰り返しすぎない
- 画面がなくても内容が分かるようにする

音声UIでは、ユーザーが情報を見返せません。

そのため、読み上げ文は短く、自然で、必要な情報に絞ることが重要だと学びました。

## Fire TV との干渉で学んだこと

Alexa Skill の検証では、想定したAlexa端末ではなく、Fire TV側が反応するような挙動も確認しました。

この経験から、音声UIでは実装だけでなく、ユーザーの利用環境によって挙動が変わる可能性があると分かりました。

音声アシスタントは、同じアカウントや同じ空間に複数端末がある場合、どの端末が反応するかが利用体験に影響します。

WebアプリではURLにアクセスすれば対象が明確ですが、音声UIでは呼び出し先や端末の選ばれ方も検証ポイントになります。

## この構成で学べたこと

このプロジェクトを通して、Alexa Skill と Lambda の基本構成だけでなく、音声UI特有の設計や外部API利用時の制約を学べました。

特に大きかった学びは以下です。

- 音声UIでは呼び出しやすさが重要
- Invocation Name は設計要素になる
- Alexa Skills Kit と Lambda の役割を分けて考える必要がある
- 外部APIは実装前に利用条件を確認する必要がある
- 複数API呼び出しでは部分失敗を考える必要がある
- Promise.all は一部失敗時の挙動に注意が必要
- Lambda のタイムアウトと応答速度が利用体験に影響する
- APIレスポンスは読み上げ用に整形する必要がある
- 技術的に作れることと、規約上利用できることは別
- 検証で断念した理由も学習成果として整理できる

## 今後の改善余地

今後改善するなら、以下を検討します。

- 天気情報の読み上げ文をさらに自然にする
- API失敗時の返答パターンを増やす
- Promise.allSettled などを使った部分成功対応
- APIごとのタイムアウト設定
- Invocation Name の改善
- スロットを使った地点指定
- ユーザーごとの地点設定
- 遅延情報APIの利用条件を再調査
- Alexa Developer Console 上のテストパターン整理
- ログ出力とエラー調査手順の整理

学習プロジェクトとしては、完成した機能だけでなく、外部API制約や音声UI特有の難しさを含めて整理することを重視しました。
