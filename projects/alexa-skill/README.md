# Alexa Skill

## 概要

Alexaスキルを題材に、音声UIとAWS Lambdaを連携させた朝の情報取得プロジェクトです。

「朝に必要な情報を、複数のアプリを開かず音声でまとめて確認する」ことを目的に、自宅と勤務地の天気を取得してAlexaに読み上げさせる仕組みを作りました。

当初は天気に加えて電車の遅延情報も読み上げる想定でしたが、個人開発で利用できる運行情報APIの制約により、実証は天気取得を中心に整理しています。

## 作成背景

朝の支度中は、天気、気温、交通情報などを確認したい一方で、スマートフォンで複数のアプリを開くのは手間です。

Alexaの標準機能でも天気確認や定型アクションはできますが、自宅と勤務地の2地点の天気や、自分が使う路線だけの情報をまとめるには制約があります。

そこで、Alexaを音声の入口として使い、Lambda側で外部APIから必要な情報を取得して、1つの読み上げ文にまとめる構成を検証しました。

## 使用技術

- Alexa Skills Kit
- AWS Lambda
- Node.js
- 外部API連携
- Open-Meteo
- CloudWatch Logs
- Lambda環境変数

## 構成

    Alexa
      ↓ カスタムスキル
    AWS Lambda
      ├─ 天気API
      └─ 遅延情報API 調査
      ↓
    読み上げ文を生成
      ↓
    Alexaが読み上げ

## 実装内容

- Alexaカスタムスキルを作成
- LambdaをAlexa Skills Kitのエンドポイントとして設定
- Lambdaから外部APIを呼び出して天気情報を取得
- 自宅と勤務地の2地点の最高気温を読み上げ文に整形
- Lambda環境変数で地点情報を管理
- CloudWatch Logsで実行ログを確認
- 開発中はInvocation Nameで起動し、実運用では定型アクションから起動する方針を検討

## 工夫した点

最初から機能を増やしすぎず、まずは「AlexaからLambdaを呼び出し、Lambdaが外部APIを叩き、結果をAlexaに返す」最小構成を作ることを優先しました。

これにより、音声UI、Lambda実行、外部API連携、読み上げレスポンス生成の流れを段階的に確認できるようにしました。

また、最終的に毎朝使うことを想定し、開発中の起動方法と実運用の起動方法を分けて考えました。開発中はInvocation Nameでスキル本体を直接呼び、実運用ではAlexaアプリの定型アクションから起動する方針にしています。

## 詰まった点

Alexa側では、Invocation Nameが初期値のままだとスキルを呼び出せないこと、Endpoint保存やLambda側のAlexa Skills Kitトリガー設定が必要なことを確認しました。

また、実機や定型アクションでは、テキストシミュレータでは動く発話が音声ではうまく認識されないことがありました。Fire TV Stickなどの関連デバイスがあると、定型アクションのカスタム文が意図しない向き先に流れることもあり、コード以外の切り分けも必要でした。

遅延情報の追加では、実装上は取得処理と読み上げ文生成を分ける方針を整理しました。一方で、個人利用可能な運行情報APIの制約が大きく、駅すぱあとAPIではフリープランで運行情報APIを利用できず、法人契約が必要であることを確認しました。

## 学び

このプロジェクトで学んだのは、音声UIでは「動くAPIを作る」だけでなく、聞き取りやすさ、起動しやすさ、情報量の調整まで含めて考える必要があるという点です。

また、外部API連携では、実装だけでなく利用条件や契約条件の確認が重要だと学びました。遅延情報の取得では、コードの問題ではなく、個人開発で利用できるデータ提供条件の制約がボトルネックになりました。

この経験から、技術検証では成功したかどうかだけでなく、なぜ成功しなかったのかを切り分けて説明できることにも価値があると感じました。
