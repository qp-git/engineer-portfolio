# Alexa Skill

## 概要

Alexa Skill を題材に、音声入力をトリガーとして AWS Lambda を呼び出し、外部 API の取得結果を音声向けに整形して返す構成を検証したプロジェクトです。

朝の支度中に必要な情報を、スマートフォンで複数のアプリを開かずに音声でまとめて確認することを目的に、自宅と勤務地の 2 地点の天気を取得して Alexa に読み上げさせる仕組みを作りました。

当初は天気に加えて電車の遅延情報も読み上げる想定でしたが、個人開発で利用できる運行情報 API の制約により、遅延情報は実装完了ではなく調査・検証結果として整理しています。

このプロジェクトでは、完成した機能だけでなく、音声 UI、Lambda 連携、外部 API 利用条件、部分失敗時の扱いまで含めて学習しました。

## プロジェクト情報

- 種別: 自主学習 / 個人開発
- 実施時期: 2026年4月頃
- 主な目的: Alexa Skill を題材に、音声 UI、AWS Lambda、外部 API 連携を検証すること

## 作成背景

朝の支度中に、天気や気温などをハンズフリーで確認できる Alexa は便利だと感じていました。一方で、標準機能だけでは思い通りの情報を取得できない場面もあり、Alexa がどのように発話を処理し、情報を取得して応答を返しているのかに関心を持ちました。

調べていく中で、Alexa Skill を自作できることを知り、音声入力をきっかけに Lambda を呼び出し、外部 API と連携して応答を返す構成を自分で検証できるのではないかと考えました。

実装では、Alexa を音声の入口として使い、Lambda 側で必要な情報を外部 API から取得し、自宅と勤務地の 2 地点の天気を 1 つの読み上げ文として返す構成を検証しました。

## 使用技術

- Alexa Skills Kit
- AWS Lambda
- Node.js
- Open-Meteo API
- CloudWatch Logs
- Lambda 環境変数

## 構成

~~~text
ユーザー
  ↓ 音声入力
Alexa 端末
  ↓
Alexa Skill
  ↓ Lambda ARN をエンドポイントとして直接呼び出し
AWS Lambda
  ├─ Open-Meteo API から天気情報を取得
  ├─ 取得結果を読み上げ文に整形
  └─ CloudWatch Logs で実行ログを確認
  ↓
Alexa が音声で読み上げ
~~~

## 実装・検証内容

- Alexa カスタムスキルを作成
- Alexa Skills Kit のエンドポイントとして Lambda ARN を指定
- API Gateway を挟まず、Alexa Skill から Lambda を直接呼び出す構成を確認
- Lambda から Open-Meteo API を呼び出し、自宅と勤務地の 2 地点の天気情報を取得
- 取得したデータを、Alexa が読み上げやすい日本語の文章に整形
- Lambda 環境変数で地点情報を管理
- CloudWatch Logs で Lambda の実行状況やエラーを確認
- 遅延情報 API の追加について、API 利用条件や実装上の課題を調査

## 実装の流れで整理したポイント

### 1. 音声入力から Lambda を呼び出す

当初は、外部から Lambda を呼び出すには API Gateway が必要だと考えていました。

しかし Alexa Skill では、Alexa Skills Kit のエンドポイントとして Lambda 関数の ARN を指定できるため、API Gateway を挟まずに構成できます。

この経験から、一般的な Web API 構成をそのまま当てはめるのではなく、サービスごとの連携方法を確認したうえで構成を選ぶ重要性を学びました。

### 2. 最小構成から段階的に確認する

最初から機能を増やしすぎず、まずは固定文を返す Lambda を作り、その後で天気 API 連携、Alexa Skill 連携へ進めました。

これにより、以下を段階的に切り分けられるようにしました。

- Lambda 単体で正常に実行できるか
- 外部 API からデータを取得できるか
- Alexa Skill から Lambda が呼び出されるか
- Lambda の戻り値を Alexa が読み上げられるか

### 3. 外部 API の結果を音声向けに整形する

音声 UI では、Web 画面のように情報を一覧表示したり、あとから見返したりできません。

そのため、API レスポンスをそのまま返すのではなく、短く自然な読み上げ文に整形する必要があります。

このプロジェクトでは、自宅と勤務地の 2 地点の情報を 1 つの文章にまとめ、朝の支度中でも聞き取りやすい長さになるよう意識しました。

### 4. 複数 API 連携時の部分失敗を考える

天気情報と遅延情報のように複数の外部 API を扱う場合、一部の API 失敗で全体を失敗させない設計が必要だと考えました。

特に、天気は朝レポートの本体、遅延情報は追加情報として扱い、遅延情報の取得に失敗しても天気情報は返せる構成を検討しました。

この検討を通じて、`Promise.all()` で複数処理をまとめると、どれか 1 つの失敗で全体が失敗する点にも注意が必要だと学びました。

### 5. API 利用条件まで含めて切り分ける

遅延情報 API の追加では、実装ミス、タイムアウト、API キー、利用プランの制約を切り分けました。

最終的には、個人利用可能な範囲では目的の運行情報を取得しにくいことが分かり、実装完了扱いにはしませんでした。

この経験から、外部サービス連携ではコードだけでなく、API の提供範囲、利用条件、契約形態まで確認する必要があると学びました。

## 学び

このプロジェクトで特に大きかった学びは、Alexa Skill が Lambda の「入口」と「出口」の両方を担えるという点です。

Alexa は、ユーザーの音声入力を受け取り、Lambda を起動するトリガーになります。さらに、Lambda の処理結果を Alexa に返すことで、ユーザーへ音声で応答できます。

もう一つの学びは、外部 API は技術的に呼び出せるかだけでなく、利用条件や提供範囲まで確認する必要があるという点です。

天気 API は比較的検証しやすかった一方で、遅延情報 API は無料プランや個人利用では扱える情報に制限がありました。

この経験から、技術検証では成功したかどうかだけでなく、なぜ成功しなかったのかを切り分けて説明できることにも価値があると感じました。

## 今後の改善余地

- 天気情報の読み上げ文をさらに自然にする
- API ごとのタイムアウト設定を整理する
- `Promise.allSettled()` などを使った部分成功対応を検討する
- API キーを Secrets Manager で管理する構成を検討する
- Invocation Name や定型アクションの設定を改善する
- ユーザーごとの地点や利用路線を保存する場合は DynamoDB などの利用を検討する
- 個人利用しやすい運行情報データソースが見つかった場合、遅延情報連携を再検証する
