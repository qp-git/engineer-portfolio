# Alexa Skill

## 概要

Alexa スキルを題材に、音声入力から AWS Lambda を呼び出し、外部 API の結果を音声で返す構成を検証したプロジェクトです。

「朝に必要な情報を、複数のアプリを開かず音声でまとめて確認する」ことを目的に、自宅と勤務地の天気を取得して Alexa に読み上げさせる仕組みを作りました。

当初は天気に加えて電車の遅延情報も読み上げる想定でしたが、個人開発で利用できる運行情報 API の制約により、実装の中心は天気取得と読み上げ文生成に絞っています。

## プロジェクト情報

* 種別: 自主学習 / 個人開発
* 実施時期: 2026年4月頃
* 主な目的: Alexa スキルを題材に、音声 UI と AWS Lambda 連携を検証すること

## 作成背景

朝の支度中は、天気、気温、交通情報などを確認したい一方で、スマートフォンで複数のアプリを開くのは手間です。

Alexa の標準機能でも天気確認や定型アクションはできますが、自宅と勤務地の 2 地点の天気や、自分が使う路線だけの情報をまとめるには制約があります。

そこで、Alexa を音声の入口として使い、Lambda 側で外部 API から必要な情報を取得して、1つの読み上げ文にまとめる構成を検証しました。

## 構成の概要

基本的な流れは以下です。

```text
ユーザー
  -> Alexa Skill
  -> AWS Lambda
  -> 外部 API
  -> AWS Lambda
  -> 読み上げ文を生成
  -> Alexa が読み上げ
```

Alexa Skill では、Alexa Skills Kit のエンドポイントとして Lambda 関数を指定できます。

そのため、このプロジェクトでは API Gateway を使わず、Alexa Skills Kit から Lambda を直接呼び出す構成を確認しました。

## 使用技術

* Alexa Skills Kit
* AWS Lambda
* Node.js
* 外部 API 連携
* Open-Meteo
* CloudWatch Logs
* Lambda 環境変数

## 実装・検証したこと

* Alexa カスタムスキルを作成
* Lambda を Alexa Skills Kit のエンドポイントとして設定
* Lambda から外部 API を呼び出して天気情報を取得
* 自宅と勤務地の 2 地点の最高気温を読み上げ文に整形
* Lambda 環境変数で地点情報を管理
* CloudWatch Logs で実行ログを確認
* 開発中は Invocation Name で起動し、実運用では定型アクションから起動する方針を検討
* 遅延情報 API の利用条件を調査し、個人開発で扱える範囲を確認

## 工夫した点

最初から機能を増やしすぎず、まずは「Alexa から Lambda を呼び出し、Lambda が外部 API を呼び出し、結果を Alexa に返す」最小構成を作ることを優先しました。

これにより、音声 UI、Lambda 実行、外部 API 連携、読み上げレスポンス生成の流れを段階的に確認できるようにしました。

また、毎朝使うことを想定し、開発中の起動方法と実運用の起動方法を分けて考えました。開発中は Invocation Name でスキル本体を直接呼び、実運用では Alexa アプリの定型アクションから起動する方針にしています。

## 詰まった点

Alexa 側では、Invocation Name が初期値のままだとスキルを呼び出せないこと、Endpoint 保存や Lambda 側の Alexa Skills Kit トリガー設定が必要なことを確認しました。

また、実機や定型アクションでは、テキストシミュレータでは動く発話が音声ではうまく認識されないことがありました。Fire TV Stick などの関連デバイスがあると、定型アクションのカスタム文が意図しない向き先に流れることもあり、コード以外の切り分けも必要でした。

遅延情報の追加では、実装上は取得処理と読み上げ文生成を分ける方針を整理しました。一方で、個人利用可能な運行情報 API の制約が大きく、無料プランや個人利用では扱える情報に限りがあることも分かりました。

## このプロジェクトで学んだこと

このプロジェクトで特に大きかった学びは、Alexa Skill が Lambda の「入口」と「出口」の両方を担えるという点です。

Alexa は、ユーザーの音声入力をきっかけに Lambda を起動するトリガーになります。また、Lambda の処理結果を Alexa に返すことで、ユーザーへ音声で応答できます。

当初は、外部から Lambda を呼び出すには API Gateway が必要だと考えていました。しかし Alexa Skill では、Alexa Skills Kit のエンドポイントとして Lambda 関数の ARN を指定できるため、API Gateway を挟まずに構成できます。

また、音声 UI では「動く API を作る」だけでなく、聞き取りやすさ、起動しやすさ、情報量の調整まで含めて考える必要があると学びました。

外部 API 連携では、技術的に呼び出せるかだけでなく、利用条件や提供範囲まで確認することも重要です。天気 API は比較的検証しやすかった一方で、遅延情報 API は個人開発で扱えるデータ提供条件が制約になりました。

この経験から、技術検証では成功した点だけでなく、なぜ実装範囲を絞ったのか、どこに外部サービス側の制約があったのかを説明できることにも価値があると感じました。
