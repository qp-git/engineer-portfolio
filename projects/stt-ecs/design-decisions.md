# STT + ECS 設計判断メモ

このプロジェクトでは、音声を文字起こし・要約する Web アプリを題材に、AI API を使うアプリケーション構成と、AWS ECS 上でのコンテナ実行基盤を検証しました。

単にアプリを作るだけでなく、ローカル実装、Docker化、ECRへの登録、ECSでの実行、ALB経由の公開、ヘルスチェック、タスク定義、ロールバックまでを一連の流れとして整理することを重視しました。

## 音声処理アプリを題材にした理由

音声処理アプリを題材にした理由は、AI API の利用と Web アプリ構成を同時に学びやすいからです。

音声ファイルを扱うアプリでは、以下の要素を一通り確認できます。

- ブラウザからのファイル送信
- バックエンドでのファイル受け取り
- 外部APIの呼び出し
- 処理結果の画面表示
- エラー時の切り分け
- APIキーなどの秘密情報管理
- 処理時間が長い場合の待ち時間やUI設計

また、STT と LLM の役割が分かれているため、AI API を使うアプリでも処理を分解して考える練習になります。

## 非リアルタイム版から始めた理由

最初からリアルタイムSTTに取り組むと、音声ストリーム、低遅延通信、途中結果の表示、発話区切りなど、考えることが一気に増えます。

そのため、まずは録音後に音声ファイルをまとめて送る非リアルタイム版から作りました。

非リアルタイム版を先に作ることで、以下を確認しやすくなります。

- ブラウザで音声を録音できるか
- 録音データをバックエンドに送信できるか
- Flask側でファイルを受け取れるか
- OpenAI APIへ文字起こし依頼を出せるか
- 文字起こし結果を要約処理へ渡せるか
- 最終結果をブラウザに返せるか

この段階で基本的な処理の流れを固めてから、リアルタイム版の検証に進む方が、問題の切り分けがしやすいと判断しました。

## Flask を使った理由

バックエンドには Python / Flask を利用しました。

Flask を選んだ理由は、音声ファイルの受け取り、外部API呼び出し、レスポンス返却という流れを小さく実装しやすいためです。

また、Python は AI API や音声処理関連のサンプルやライブラリが多く、学習プロジェクトとして検証しやすい利点があります。

Flask を使うことで、以下をシンプルに確認できます。

- HTTPリクエストの受け取り
- ファイルアップロード処理
- 環境変数の利用
- 外部APIの呼び出し
- JSONレスポンスの返却
- Dockerコンテナ化

大規模なWebアプリケーションフレームワークを使うよりも、今回の検証範囲では Flask の方が処理の流れを把握しやすいと判断しました。

## STT と LLM を分けて考えた理由

このアプリでは、音声を直接要約するのではなく、まず音声をテキストに変換し、その後でテキストを要約します。

処理を分けることで、以下を説明しやすくなります。

- STT は音声を文字に変換する役割
- LLM はテキストを要約・整理する役割
- 文字起こし結果が悪いと、要約品質にも影響する
- どの段階で失敗しているのか切り分けやすい

AI API を使うアプリでは、「AIを呼ぶ」と一括りにせず、入力、変換、加工、出力のどこにAIを使っているかを整理することが重要だと考えました。

## Docker化した理由

ローカルで動く Flask アプリを、そのままECSに載せることはできません。

ECSで動かすには、アプリケーションをコンテナとして扱える形にする必要があります。

そのため、Flask アプリを Docker イメージ化しました。

Docker化した理由は以下です。

- 実行環境を固定できる
- Pythonライブラリの依存関係をまとめられる
- ローカルとECSで実行方法を近づけられる
- ECRにイメージを登録できる
- ECSタスク定義から参照できる
- コンテナ実行基盤の理解につながる

Docker化により、アプリケーションそのものと実行環境をひとまとまりとして扱えるようになります。

## ECS を使った理由

このプロジェクトでは、単にコンテナを作るだけでなく、AWS上でコンテナをどう運用するかを理解することを目的にしました。

そのため、ECSを利用しました。

ECSを使うことで、以下を確認できます。

- ECRに保存したイメージを使ってタスクを起動する流れ
- タスク定義、タスク、サービスの関係
- desired tasks によるタスク数の維持
- ALBとの連携
- ヘルスチェック
- タスク定義 revision
- ロールバックの考え方
- Fargate と ECS on EC2 の違い

ECSは、コンテナ実行そのものだけでなく、周辺のネットワーク、ロードバランシング、ログ、秘密情報管理も含めて考える必要があるため、インフラ学習の題材として有効だと感じました。

## Fargate と ECS on EC2 の両方を調べた理由

ECSには、主に Fargate と ECS on EC2 の2つの実行方式があります。

Fargate は、コンテナを載せるサーバーの管理をAWSに任せられるため、ECSの基本構造を理解しやすいです。

一方、ECS on EC2 では、コンテナを配置するEC2インスタンスのリソースやAuto Scaling Groupも意識する必要があります。

両方を調べた理由は以下です。

- FargateではECSの基本構造を理解しやすい
- ECS on EC2では配置先リソースの考え方を学べる
- desired tasks と EC2台数の違いを理解できる
- 実務ではコストや制約に応じて方式が変わる可能性がある
- コンテナを「どこで動かすか」という視点を持てる

この比較により、ECSは単にタスクを起動するだけではなく、実行基盤の選択も設計判断の一部だと理解しました。

## ALB を利用した理由

ECSタスクを外部から利用するには、入口が必要です。

このプロジェクトでは ALB を利用し、外部からのHTTPリクエストをECSタスクへ振り分ける構成を検証しました。

ALBを利用した理由は以下です。

- 外部公開の入口として利用できる
- 複数タスクへの振り分けを考えられる
- ヘルスチェックによって正常なタスクだけに通信できる
- ECSサービスとの連携を学べる
- 実務に近い構成を確認できる

特に重要だったのは、ECSタスクが Running でも、ALBのヘルスチェックに通らなければ利用者からはアクセスできないという点です。

この違いを理解することで、障害時の切り分けがしやすくなります。

## ヘルスチェックを重視した理由

ECSでは、タスクが起動していることと、アプリケーションが正常に応答できることは別です。

そのため、ALBのヘルスチェックを重視しました。

ヘルスチェックでは、以下を確認する必要があります。

- アプリケーションが起動しているか
- 指定ポートで待ち受けているか
- ヘルスチェック用パスに応答できるか
- ステータスコードが期待通りか
- ターゲットグループ上で healthy になっているか

この観点は、アプリケーションのデプロイだけでなく、運用時の監視や障害対応にもつながると感じました。

## タスク定義 revision を意識した理由

ECSのタスク定義は、変更するたびに revision が増えます。

この仕組みを意識することで、変更履歴とロールバックの考え方を学べます。

たとえば、以下のような変更はタスク定義の revision として管理されます。

- コンテナイメージの変更
- CPU / メモリの変更
- ポートマッピングの変更
- 環境変数の変更
- Secrets参照の変更
- ログ設定の変更

問題が起きたときに、どの revision から挙動が変わったのかを確認できることは重要です。

このプロジェクトでは、ECSの revision を単なる番号ではなく、運用上の変更履歴として捉えることを意識しました。

## APIキーをアプリ外で管理する理由

OpenAI APIを使うにはAPIキーが必要です。

APIキーをGitHubやDockerイメージに含めてしまうと、漏えいリスクがあります。

そのため、以下のような扱いを前提にしました。

- APIキーをGitHubに載せない
- Dockerイメージ内に埋め込まない
- ローカルでは環境変数を利用する
- AWS上ではSecrets Managerなどの利用を検討する
- ECSタスク定義から秘密情報を参照する

AI APIを使うアプリでは、APIキーがサービス利用権限そのものに近い意味を持ちます。

そのため、アプリケーションの実装だけでなく、秘密情報管理も設計の一部として考える必要があります。

## リアルタイムSTTを別検証にした理由

リアルタイムSTTは、非リアルタイム版とは構成が大きく異なります。

非リアルタイム版では音声ファイルをまとめて送りますが、リアルタイム版では音声を送りながら結果を受け取ります。

リアルタイム版で追加される論点は以下です。

- 音声を小さく区切って送る
- WebRTC や Data Channel を扱う
- 途中結果と確定結果を区別する
- 低遅延で画面に反映する
- 発話区切りをどう扱うか考える
- 日本語の文脈や区切りを考慮する

このため、リアルタイムSTTは非リアルタイム版の延長ではなく、別の検証テーマとして扱いました。

段階を分けることで、基本構成とリアルタイム通信の難しさを混同せずに整理できます。

## ロールバックを意識した理由

ECSでは、タスク定義 revision やコンテナイメージタグを通じて、以前の状態へ戻すことを検討できます。

個人開発でも、変更後にアプリが起動しない、ヘルスチェックに失敗する、API呼び出しが失敗するなどの問題は起こり得ます。

そのため、最初から以下を意識しました。

- 変更前の状態をGitで残す
- Dockerイメージの更新履歴を意識する
- タスク定義 revision を確認する
- 変更後にヘルスチェックを見る
- 問題が起きた場合は前の revision に戻すことを検討する

実務では、デプロイできることだけでなく、問題が起きたときに戻せることも重要です。

このプロジェクトでは、その考え方をECSの学習に含めました。

## この構成で学べたこと

このプロジェクトを通して、AI APIを使うWebアプリとAWSコンテナ基盤の両方を学べました。

特に大きかった学びは以下です。

- AI APIを使うアプリでも、Webアプリの基本構成は変わらない
- STT と LLM は役割を分けて考える必要がある
- Docker化はECSへ載せる前提として重要
- ECSではタスク定義、タスク、サービスの関係が重要
- Fargate と ECS on EC2 では意識する範囲が違う
- ALBのヘルスチェックは障害切り分けに重要
- desired tasks と Auto Scaling Group は管理対象が違う
- APIキーなどの秘密情報はアプリ外で管理すべき
- revision やロールバックは運用上重要

## 今後の改善余地

今後改善するなら、以下を検討します。

- GitHub ActionsによるECR pushとECS更新の自動化
- Secrets Managerとの連携強化
- CloudWatch Logsでのログ確認
- ALBアクセスログの確認
- ヘルスチェック専用エンドポイントの整備
- 音声ファイルサイズ制限の設計
- 長時間処理時のUI改善
- 非同期処理やジョブキューの検討
- リアルタイムSTT版のUI改善
- 費用見積もりと運用コストの整理

学習プロジェクトとしては、まず一連の構成要素を自分でつなぎ、どこで何が起きているのか説明できる状態にすることを重視しました。
