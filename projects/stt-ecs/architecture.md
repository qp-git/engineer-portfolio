# STT + ECS 構成メモ

このプロジェクトは、ブラウザで録音した音声をアプリケーションサーバーへ送信し、OpenAI API を使って文字起こしと要約を行う Web アプリを、Docker 化して AWS ECS 上で動かす検証プロジェクトです。

アプリケーションの構成だけでなく、ECS、ECR、ALB、タスク定義、サービス、Fargate、ECS on EC2 など、コンテナ実行基盤として必要になる周辺要素も整理しています。

## アプリケーション全体の流れ

非リアルタイム版では、ブラウザで録音した音声ファイルを Flask アプリへ送信し、バックエンド側から OpenAI API を呼び出します。

    ブラウザ
      ↓ 音声ファイル送信
    Flask アプリケーション
      ↓ 文字起こし依頼
    OpenAI API
      ↓ 文字起こし結果
    Flask アプリケーション
      ↓ 要約依頼
    OpenAI API
      ↓ 要約結果
    Flask アプリケーション
      ↓ 結果返却
    ブラウザ

この構成では、AI 処理そのものをアプリ内で実行しているわけではありません。

ブラウザで入力を受け取り、Flask が外部 API を呼び出し、その結果を画面に返す構造です。

## 使用技術

| 分類 | 技術・サービス | 役割 |
|---|---|---|
| フロントエンド | HTML / JavaScript | 録音、音声送信、結果表示 |
| バックエンド | Python / Flask | 音声ファイル受信、API呼び出し、結果返却 |
| AI API | OpenAI API | 文字起こし、要約 |
| コンテナ | Docker | アプリケーション実行環境のパッケージ化 |
| イメージ管理 | Amazon ECR | Dockerイメージの保存 |
| コンテナ実行 | Amazon ECS | コンテナの起動・維持 |
| 実行方式 | Fargate / ECS on EC2 | コンテナの実行基盤 |
| ロードバランサー | ALB | 外部からの入口、ヘルスチェック |
| 秘密情報管理 | Secrets Manager など | APIキーなどの管理候補 |

## ローカルアプリ構成

最初は、ローカル環境で Flask アプリとして動作確認しました。

    ブラウザ
      ↓
    Flask
      ↓
    OpenAI API

この段階では、以下を確認しています。

- ブラウザで録音できること
- 音声ファイルを Flask に送信できること
- Flask から OpenAI API を呼び出せること
- 文字起こし結果を取得できること
- 文字起こし結果を要約に渡せること
- 結果をブラウザへ返せること

この段階で、AI を使う Web アプリでも基本構造は「入力を受ける」「バックエンドで処理する」「結果を返す」であると整理しました。

## 非リアルタイム処理の構成

非リアルタイム版では、録音が終わったあとに音声ファイル全体を送信します。

    録音開始
      ↓
    録音終了
      ↓
    音声ファイルを送信
      ↓
    STTで文字起こし
      ↓
    LLMで要約
      ↓
    結果を表示

この方式はリアルタイム性は低いものの、処理の流れが単純で、最初の実装・検証に向いています。

特に、以下の点を確認しやすい構成です。

- 音声ファイルの送信
- Flask 側のファイル受け取り
- API呼び出しの順序
- 文字起こしと要約の役割分担
- エラー発生箇所の切り分け

## STT と LLM の役割分担

このアプリでは、音声を直接要約しているのではなく、段階的に処理しています。

1. 音声を STT に渡してテキスト化する
2. テキスト化された内容を LLM に渡して要約する
3. 要約結果をブラウザへ返す

STT は Speech To Text の略で、音声を文字に変換する技術です。

LLM は、得られたテキストをもとに要約や整理を行います。

この役割分担を意識することで、AI API を使ったアプリケーションでも、どこで何を処理しているのかを説明しやすくなります。

## Docker化後の構成

Flask アプリを Docker イメージとして固めることで、ローカル環境に依存しにくい形にしました。

    Flask アプリ
      ↓
    Docker イメージ
      ↓
    ECR
      ↓
    ECS タスク

Docker化により、以下を整理できます。

- アプリケーションの実行環境
- Pythonライブラリの依存関係
- 起動コマンド
- 公開ポート
- ECS上で動かすための単位

ECS では、Docker イメージを直接指定するのではなく、タスク定義を通して利用します。

## ECS 全体構成

ECS では、ECR に保存したイメージをもとにタスクを起動し、サービスで必要数を維持します。

    ECR
      ↓ イメージ参照
    タスク定義
      ↓ 起動条件
    タスク
      ↓ 維持
    ECS サービス
      ↓ 振り分け
    ALB

主な関係は以下です。

| 要素 | 役割 |
|---|---|
| ECR | Docker イメージを保存する場所 |
| タスク定義 | どのイメージを、どのCPU・メモリ・ポートで動かすかを定義する |
| タスク | タスク定義に基づいて実際に起動したコンテナ実行単位 |
| サービス | 指定した数のタスクを維持する |
| ALB | 外部からのアクセスを受け、正常なタスクに振り分ける |

## Fargate 構成

最初に Fargate を使うと、EC2 インスタンスの管理を意識しすぎずに ECS の基本構造を確認できます。

    利用者
      ↓
    ALB
      ↓
    ECS Service
      ↓
    Fargate Task
      ↓
    Flask Container

Fargate では、コンテナを載せるサーバー側の管理を AWS に任せられます。

そのため、初期段階では以下を理解しやすくなります。

- ECR にイメージを置く
- タスク定義を作る
- タスクを起動する
- サービスで必要数を維持する
- ALB のヘルスチェックを通す

ECS の用語や流れをつかむには、Fargate から入る方が整理しやすいと感じました。

## ECS on EC2 構成

ECS on EC2 では、コンテナを配置する EC2 インスタンスも意識する必要があります。

    利用者
      ↓
    ALB
      ↓
    ECS Service
      ↓
    ECS Container Instance
      ↓
    EC2 上の Docker Container

Fargate ではあまり意識しなかった、以下のような点が重要になります。

- EC2 の CPU
- EC2 のメモリ
- Docker や ECS Agent が使うリソース
- タスク配置に必要な空きリソース
- ポート競合
- Auto Scaling Group との関係

この段階で、ECS では「コンテナを起動する」だけでなく、「どこに配置できるか」も重要だと学びました。

## ALB とヘルスチェック

ALB は外部からの入口として機能し、正常なタスクへリクエストを振り分けます。

    利用者
      ↓
    ALB
      ↓ ヘルスチェックOKのターゲットへ転送
    ECS Task

ここで重要なのは、ECS 側でタスクが起動していても、ALB のヘルスチェックに通っていなければ利用者からの通信は流れない点です。

つまり、以下は別々に確認する必要があります。

- ECS タスクが起動しているか
- コンテナ内のアプリが起動しているか
- 指定ポートで待ち受けているか
- ALB のヘルスチェックに通っているか
- ターゲットグループで healthy になっているか

ECS 側の Running と、ALB 側の Healthy は同じ意味ではありません。

## タスク定義と revision

ECS のタスク定義は、変更するたびに revision が増えます。

    task-definition:1
      ↓ 修正
    task-definition:2
      ↓ 修正
    task-definition:3

タスク定義には、以下のような情報を含めます。

- 使用するコンテナイメージ
- CPU / メモリ
- ポートマッピング
- 環境変数
- Secrets
- ログ設定
- 起動コマンド

revision があることで、変更履歴をたどりやすくなり、問題が起きたときに前の revision へ戻すことも検討できます。

この点は、単にアプリを動かすだけでなく、運用やロールバックを考える上で重要だと感じました。

## desired tasks と Auto Scaling Group の違い

ECS サービスの desired tasks は、「アプリケーションのタスクを何個動かしたいか」を表します。

一方、Auto Scaling Group は、「コンテナを載せる EC2 インスタンスを何台用意したいか」を表します。

| 項目 | 管理対象 |
|---|---|
| ECS desired tasks | アプリケーションのタスク数 |
| Auto Scaling Group | EC2 インスタンス数 |

ECS on EC2 では、この2つを混同すると構成が分かりにくくなります。

アプリのタスク数を増やしても、配置できる EC2 側のリソースが足りなければ起動できません。

逆に EC2 を増やしても、desired tasks が増えなければアプリのタスク数は増えません。

## Secrets の扱い

OpenAI API を利用するため、APIキーの扱いには注意が必要です。

ローカルでは環境変数で扱うとしても、ECS 上では以下のような方法を検討します。

- Secrets Manager に保存する
- ECS タスク定義から参照する
- GitHub に API キーを載せない
- Docker イメージ内に秘密情報を埋め込まない

AI API を使うアプリでは、APIキーが実質的な認証情報になります。

そのため、アプリケーションの実装だけでなく、インフラ側での秘密情報管理も重要になります。

## リアルタイム版の検証

非リアルタイム版に加えて、リアルタイム STT の検証も行いました。

リアルタイム版では、録音後にまとめて送るのではなく、音声を分割して送りながら文字起こし結果を受け取る構成になります。

    ブラウザ
      ↓ 音声ストリーム
    WebRTC / Data Channel
      ↓
    Realtime API
      ↓
    transcription event
      ↓
    ブラウザ

リアルタイム版では、非リアルタイム版とは別の難しさがあります。

- 音声を小さく区切って送る必要がある
- 低遅延で結果を受け取る必要がある
- 発話の区切りをどう扱うか考える必要がある
- 日本語では区切りや文脈の扱いが難しい
- UI側で途中結果と確定結果をどう表示するか考える必要がある

この検証により、同じ STT でも「ファイルを送って結果を待つ方式」と「リアルタイムに結果を受け取る方式」では、構成と考えることが変わると分かりました。

## 構成上の学び

このプロジェクトでは、AI API を使ったアプリケーション開発と、AWS 上でのコンテナ実行基盤をあわせて検証しました。

特に学びが大きかった点は以下です。

- AI APIを使うアプリでも、Webアプリの基本構成は変わらない
- STT と LLM は役割が異なる
- Docker化すると、アプリをECSに載せる流れが見えやすくなる
- ECSではタスク定義、タスク、サービスの関係が重要
- Fargate と ECS on EC2 では意識する範囲が変わる
- ALB のヘルスチェックはタスク起動とは別に確認する必要がある
- desired tasks と ASG は管理対象が違う
- APIキーなどの秘密情報はアプリ外で管理すべき
