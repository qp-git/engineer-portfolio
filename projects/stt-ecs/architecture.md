# STT + ECS アーキテクチャ

## このドキュメントの目的

このドキュメントでは、STT + ECS プロジェクトのアプリケーション処理フローと、AWS 上でのコンテナ実行構成を整理します。

README ではプロジェクト概要を、[設計判断メモ](./design-decisions.md) では設計理由を説明しています。ここでは、ブラウザ、Flask、OpenAI API、Docker、ECR、ECS、ALB がどのように関係するかを中心にまとめます。

## 全体構成

このプロジェクトは、録音した音声を文字起こしして要約する Web アプリを Docker コンテナ化し、Amazon ECS 上で動かす構成を検証したものです。

アプリケーション側の基本的な流れは以下です。

    Browser
      ↓ 音声ファイル送信
    Flask App
      ↓ 文字起こし依頼
    OpenAI API
      ↓ 文字起こし結果
    Flask App
      ↓ 要約依頼
    OpenAI API
      ↓ 要約結果
    Flask App
      ↓ 結果表示
    Browser

AWS 上では、Flask アプリを Docker イメージ化し、ECR に保存したイメージを ECS で実行します。

    Browser
      ↓
    ALB
      ↓
    ECS Service
      ↓
    ECS Task
      ↓
    Flask Container
      ↓
    OpenAI API

## 使用技術・サービスと役割

### Browser

音声を録音し、Flask アプリへ送信します。

また、文字起こし結果と要約結果を画面に表示します。

### Flask App

音声ファイルを受け取り、OpenAI API に文字起こしを依頼します。

文字起こしされたテキストを要約用モデルに渡し、要約結果をブラウザへ返します。

### OpenAI API

STT（Speech to Text）による文字起こしと、文字起こし結果の要約に利用します。

アプリケーション内に AI モデルを持つのではなく、外部 API を呼び出して処理結果を受け取る構成です。

### Docker

Flask アプリと実行環境を Docker イメージとしてまとめます。

アプリケーションコードだけでなく、Python 実行環境や依存ライブラリも含めてコンテナ化します。

### Amazon ECR

作成した Docker イメージを保存するリポジトリです。

ECS タスクは、ECR に保存されたイメージを取得してコンテナを起動します。

### Amazon ECS

Docker コンテナを AWS 上で実行するために利用します。

このプロジェクトでは、Fargate と ECS on EC2 の両方を確認しました。

### Application Load Balancer

ブラウザからのリクエストを受け取り、ECS タスクへ振り分けます。

ヘルスチェックにより、正常なタスクにだけ通信を流す役割も持ちます。

## アプリケーション処理フロー

非リアルタイム版では、録音した音声ファイルをまとめて送信し、文字起こしと要約を順番に行います。

1. ブラウザで音声を録音する
2. 録音データを Flask アプリへ送信する
3. Flask アプリが音声ファイルを受け取る
4. OpenAI API へ文字起こしを依頼する
5. 文字起こし結果を要約モデルへ渡す
6. 要約結果をブラウザへ返す

この構成により、ブラウザ、バックエンド、外部 API の役割分担を確認できます。

## Docker / ECS 実行構成

AWS 上で動かすために、Flask アプリを Docker イメージ化し、ECR に保存しました。

ECS では、タスク定義に以下のような情報を設定します。

- 利用する Docker イメージ
- コンテナのポート
- CPU / メモリ
- 環境変数
- Secrets
- ログ出力先

ECS サービスは、指定したタスク数を維持し、必要に応じて新しいタスクを起動します。

## Fargate構成

Fargate では、コンテナを実行するサーバー管理を AWS に任せます。

この構成では、ECR、タスク定義、ECS サービス、ALB といった ECS の基本要素を確認しやすくなります。

    ECR
      ↓
    ECS Task Definition
      ↓
    ECS Service
      ↓
    Fargate Task
      ↓
    Flask Container

Fargate では EC2 インスタンスを直接管理しないため、まず ECS の基本的な流れを確認するのに適しています。

## ECS on EC2構成

ECS on EC2 では、コンテナを配置する EC2 インスタンスも構成の一部になります。

タスク数を増やしても、配置先となる EC2 インスタンス側の CPU、メモリ、ポートに余裕がなければタスクは配置できません。

    Auto Scaling Group
      ↓
    EC2 Instance
      ↓
    ECS Agent
      ↓
    ECS Task
      ↓
    Flask Container

この構成では、ECS サービスが維持するタスク数と、Auto Scaling Group が用意する EC2 インスタンス数を分けて考える必要があります。

## ALB / ヘルスチェック

ALB は、外部からの HTTP リクエストを受け取り、ECS タスクへ振り分けます。

ECS タスクが Running であっても、ALB のターゲットグループ上で healthy になっていなければ、利用者からは正常にアクセスできません。

そのため、デプロイ後は以下を分けて確認します。

- ECS タスクが起動しているか
- コンテナ内の Flask アプリが指定ポートで待ち受けているか
- ヘルスチェック用パスに応答しているか
- ターゲットグループ上で healthy になっているか

この切り分けにより、ECS 側の問題なのか、アプリケーション側の問題なのか、ALB 側の問題なのかを確認しやすくなります。

## Secrets / 環境変数

OpenAI API を利用するため、API キーをアプリケーションから参照できるようにする必要があります。

ただし、API キーをコードや Docker イメージに含めるのではなく、環境変数や Secrets Manager などを通してコンテナへ渡す構成を意識しました。

Secrets Manager を利用する場合は、シークレットを作成するだけでなく、ECS タスクがそのシークレットを参照できる IAM 権限も必要になります。

## リアルタイムSTT検証の構成

録音後に音声ファイルをまとめて送る方式に加えて、リアルタイム STT も検証しました。

リアルタイム STT では、WebRTC 接続、Data Channel 接続、transcription イベントの受信までは確認しました。

一方で、低遅延で結果を返すことと、日本語として読みやすい文字起こし結果を得ることの両立が課題になりました。

この検証により、音声アプリでは API に接続できることと、実用的な品質で使えることは別の課題だと整理できます。
