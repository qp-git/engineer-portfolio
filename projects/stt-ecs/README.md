# STT + ECS

## 概要

録音した音声を文字起こしして要約するWebアプリを作成し、そのアプリをコンテナ化してAWS ECS関連サービスで動かす検証プロジェクトです。

最初に、ブラウザで録音した音声をFlaskバックエンドへ送り、OpenAI APIで文字起こしと要約を行うアプリを作成しました。その後、アプリをコンテナ化し、ECR、ECS、ALBなどを使ってAWS上で動かす構成を検証しました。

## 作成背景

会議メモや思いつきの記録など、音声をあとから文字起こしして要約できる仕組みに関心があり、自分でWebアプリとして再現しました。

また、AIを使うアプリが実際のインフラ上ではどのような構成になるのかを確認するため、ローカル実装だけで終わらせず、コンテナ化してAWS上で動かすところまで検証しました。

## 構成

    Browser
      ↓ 音声ファイル送信
    Flask App
      ↓
    OpenAI API
      ├─ 文字起こし
      └─ 要約
      ↓
    Flask App
      ↓
    Browserに結果表示

AWS上では、作成したアプリをコンテナ化し、ECRに保存したイメージをECSで実行する構成を検証しました。

## 実装内容

- ブラウザで音声を録音して送信する画面を作成
- Flaskで音声ファイルを受け取り、OpenAI APIへ渡すバックエンドを作成
- STTで音声をテキスト化し、その結果を要約モデルへ渡す処理を実装
- アプリをDockerコンテナ化
- ECRにコンテナイメージを保存
- ECS Fargateでタスクを起動
- ECS on EC2でタスクの配置やリソース制約を確認
- ALBを前段に置き、ブラウザからアクセスする入口を整理
- タスク定義のrevisionとロールバックの考え方を確認

## 工夫した点

AIを使うWebアプリといっても、アプリの中にAIそのものがいるわけではなく、ブラウザ、バックエンド、外部APIが役割分担する構成であることを意識しました。

また、ECSでは最初にFargateで全体像をつかみ、その後ECS on EC2で「タスクをどこに置くか」「EC2のメモリやポートが足りるか」といった配置の考え方を確認しました。

ブラウザで音声を扱うため、HTTPSの必要性も意識しました。ALBを前段に置くことで、利用者から見た入口を安定させ、ヘルスチェックやタスク切り替えを確認しやすくしました。

## 詰まった点

リアルタイムSTTでは、WebRTC接続、Data Channel接続、transcriptionイベント受信までは確認できましたが、日本語の精度と低遅延の両立が難しいと感じました。

ECS on EC2では、t3.microのような小さいインスタンスではOS、Docker、ECS Agentが先にリソースを使うため、タスクを置く余裕が少ないことを確認しました。アプリが落ちる以前に、そもそも配置条件を満たせず起動できない場合があると理解しました。

また、desired tasksを増やすことと、EC2台数を増やすことは別であり、必要に応じてASGで置き場そのものを増やす必要があると学びました。

## 学び

このプロジェクトでは、AIを使うアプリの基本構造と、それをAWS上で動かすためのコンテナ基盤の両方を学びました。

特に、STTとLLMは役割が異なり、音声をそのまま要約しているのではなく、まず音声を文字起こしし、そのテキストを要約モデルへ渡していることを整理できました。

ECSでは、タスク定義、タスク、サービス、desired tasks、ASG、ALB、revisionといった用語の関係を、実際に触りながら理解しました。

この経験から、アプリを動かすにはコードだけでなく、配置、入口、リソース、シークレット管理、ロールバックまで含めて考える必要があると学びました。
