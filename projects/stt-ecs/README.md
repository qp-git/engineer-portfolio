# STT + ECS

## 概要

録音した音声を文字起こしして要約するWebアプリを作成し、そのアプリをDockerコンテナ化してAmazon ECS上で動かす検証プロジェクトです。

ブラウザで録音した音声をFlaskバックエンドへ送り、OpenAI APIで文字起こしと要約を行います。その後、作成したアプリをDockerイメージ化し、ECR、ECS、ALBなどを使ってAWS上で動かす構成を検証しました。

## 作成背景

会議メモや思いつきの記録など、音声をあとから文字起こしして要約できる仕組みに関心があり、自分でWebアプリとして再現しました。

また、DockerやECSが現場でどのように使われるのかに関心があったため、ローカルで動くアプリを作って終わりにせず、コンテナ化してAWS上で実行するところまで検証しました。

その過程で、OpenAI APIを利用するアプリでは、APIキー管理、Secrets Manager、IAM権限、HTTPS化、課金リスクなど、AI APIを運用環境に載せる上で必要になる追加の設計観点も学びました。

## 構成の概要

アプリケーションの基本的な流れは以下です。

    Browser
      ↓ 音声ファイル送信
    Flask App
      ↓ 文字起こし依頼
    OpenAI API
      ↓ 文字起こし結果
    Flask App
      ↓ 要約依頼
    OpenAI API
      ↓ 要約結果
    Flask App
      ↓ 結果表示

AWS上では、FlaskアプリをDockerイメージ化し、ECRに保存したイメージをECSで実行する構成を検証しました。

## 実装・検証したこと

### アプリケーション側

- ブラウザで音声を録音して送信する画面を作成
- Flaskで音声ファイルを受け取り、OpenAI APIへ渡すバックエンドを作成
- STTで音声をテキスト化し、その結果を要約モデルへ渡す処理を実装
- 音声入力、文字起こし、要約、結果表示の流れを確認

### コンテナ・AWS側

- アプリをDockerコンテナ化
- ECRにコンテナイメージを保存
- ECS Fargateでタスクを起動
- ECS on EC2でタスク配置やリソース制約を確認
- ALBを前段に置き、ブラウザからアクセスする入口を整理
- タスク定義のrevisionとロールバックの考え方を確認

## このプロジェクトで学んだこと

このプロジェクトでは、Docker化したアプリをAmazon ECSで動かすために、アプリケーション実装だけでなく、コンテナ実行基盤や外部公開の入口まで含めて整理しました。

特に以下の点を学びました。

- Docker化したアプリをECSで動かすには、ECR、タスク定義、ECSサービス、ALBまで含めて考える必要があること
- FargateとECS on EC2では、コンテナを動かす基盤をどこまで意識するかが異なること
- ECS on EC2では、タスク数だけでなく、配置先となるEC2インスタンス側のCPU・メモリ・ポートも考える必要があること
- OpenAI APIを使う場合、APIキーをコードやDockerイメージに含めず、Secrets Managerや環境変数で扱う必要があること
- Secrets Managerを利用する場合、ECSタスクがシークレットを参照できるIAM権限も設計対象になること
- ブラウザでマイク入力や音声データを扱う場合、HTTPS化も実用上重要になること
- タスク定義のrevisionやALBのヘルスチェックを確認することで、デプロイ後の切り戻しや障害切り分けを考えやすくなること

この経験から、アプリをAWS上で動かすには、コードだけでなく、コンテナ実行基盤、リソース配置、外部公開、秘密情報管理、IAM権限、ロールバックまで含めて考える必要があると学びました。
